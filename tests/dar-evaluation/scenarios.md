# DAR 评估测试场景

> 5 个复杂企业级测试场景，每个 ≥200 字，覆盖多节点联动的完整操作流程。
> 用于基准（无 DAR）vs 增强（有 DAR）对比测试。

## 场景 1: 学术论文文献综述（paper）

**背景**：某高校研究团队正在撰写一篇关于"大语言模型在医疗诊断中应用"的综述论文。团队需要完成从文献检索、引用验证、撤稿检查到最终参考文献列表输出的完整流程。论文目标是投稿至 IEEE TPAMI 或类似顶刊。

**操作流程（多节点联动）**：
1. **节点 A — 文献检索**：在 Google Scholar 和 Semantic Scholar 上检索 2020-2025 年间关于 LLM 医疗诊断的论文，需覆盖实证研究、综述文章和预印本。需要筛选被引 >50 的高影响力论文。
2. **节点 B — 引用验证**：对检索到的每篇论文，通过 CrossRef 验证 DOI 的正确性，确保引用信息（作者、标题、期刊、年份）准确无误。发现 3 篇论文的 DOI 存在重定向问题。
3. **节点 C — 撤稿检查**：通过 Retraction Watch 检查是否有已撤稿论文混入文献列表。发现 1 篇 2022 年发表的论文因数据造假于 2024 年被撤稿，需从引用列表中移除并标注。
4. **节点 D — 时效性筛选**：检查所有引用论文的发表年份，将超过 10 年的实证研究标注为"经典但需检查是否有新研究推翻"。发现 2 篇 2013 年的论文需要补充 2024 年的新研究作为对照。
5. **节点 E — 最终输出**：生成符合 IEEE 引用格式的参考文献列表，每条引用附带 DOI、被引次数和可信度分级（T1/T2）。

**验证标准**：所有 DOI 通过 CrossRef 验证；无撤稿论文混入；引用格式 100% 符合 IEEE 规范；时效性标注完整。

## 场景 2: 安全漏洞调查（coding）

**背景**：某企业的 FastAPI 后端服务在安全扫描中发现使用了存在已知漏洞的依赖包。开发团队需要快速调查漏洞详情、评估影响范围、找到修复方案并完成升级。生产环境运行着 Django + Celery + Redis 架构，涉及多个服务的依赖链。

**操作流程（多节点联动）**：
1. **节点 A — 漏洞识别**：在 CVE 和 NVD 数据库中搜索 `python-multipart>=0.0.1,<0.0.12` 的已知漏洞。发现 CVE-2024-24762（DoS 漏洞，CVSS 7.5）。
2. **节点 B — 影响分析**：检查项目依赖树，确定哪些组件依赖了受影响的包。发现 FastAPI 和 Django 项目都间接依赖了 python-multipart。
3. **节点 C — 修复方案**：在 Snyk 漏洞库和 GitHub Security Advisories 中查找修复建议。确认升级到 0.0.12+ 可修复该漏洞。
4. **节点 D — 兼容性验证**：查阅 FastAPI 官方文档和 Django 文档，确认升级 python-multipart 不会导致 breaking change。检查 Celery 和 Redis 的兼容性。
5. **节点 E — 升级执行**：执行 `pip install --upgrade python-multipart`，运行测试套件验证无回归。更新 requirements.txt 并提交。

**验证标准**：CVE 编号准确；CVSS 评分来源为 NVD；修复版本号来自官方文档；测试全部通过。

## 场景 3: 模型选型与 Benchmark 验证（agent-builder）

**背景**：某 AI 创业公司需要为客服智能体选择最优的 LLM。团队需要在 GPT-4o、Claude-3.5-Sonnet、Llama-3.1-70B 之间做出选择。选择标准包括：推理能力、工具调用能力、成本、延迟和多语言支持。公司预算有限，需要在性能和成本之间取得平衡。

**操作流程（多节点联动）**：
1. **节点 A — Benchmark 查询**：在 Open LLM Leaderboard 和 LMSYS Chatbot Arena 查询三个模型的最新排名和评分。获取 MMLU、HumanEval、MT-Bench 等基准测试分数。
2. **节点 B — API 文档查阅**：查阅 OpenAI、Anthropic 和 Google AI 的官方 API 文档，确认工具调用（function calling）的接口差异、token 限制和定价。
3. **节点 C — 框架兼容性验证**：查阅 LangChain 和 LlamaIndex 官方文档，确认三个模型是否支持当前使用的框架版本。发现 Llama-3.1 需要升级 LangChain 到 0.2.0+。
4. **节点 D — 成本估算**：基于客服场景的平均对话长度（500 token 输入 + 200 token 输出）和日均 10000 次调用，计算三个模型的月度成本。
5. **节点 E — 延迟测试**：通过 tau-Bench 查询三个模型在工具调用场景下的延迟数据。
6. **节点 F — 最终选型报告**：综合性能、成本、延迟输出选型报告，附 benchmark 数据来源和日期。

**验证标准**：Benchmark 数据来自权威榜单；API 定价来自官方文档；所有数据标注日期；多维度对比表格完整。

## 场景 4: 事实核查与多源验证（conversation）

**背景**：用户在社交媒体上看到一条消息："2024 年中国 GDP 超过美国，成为全球第一"。用户需要验证这条信息的准确性，并了解 GDP 的不同统计口径（名义 GDP vs 购买力平价 GDP）。

**操作流程（多节点联动）**：
1. **节点 A — 权威数据查询**：在世界银行和 IMF 官方数据库中查询 2024 年中国和美国的名义 GDP 和 PPP GDP 数据。
2. **节点 B — 统计口径辨析**：查阅经济学权威资料，理解名义 GDP（按汇率换算）和购买力平价 GDP（PPP）的区别。确认中国在 PPP GDP 上确实已超过美国，但名义 GDP 仍未超过。
3. **节点 C — 事实核查**：在 Snopes 或 FactCheck.org 上搜索该传言的核查结果。确认"2024 年中国名义 GDP 超过美国"是错误信息。
4. **节点 D — 数据标注**：输出结果时，标注数据来源（World Bank/IMF）、数据年份（2024）、统计口径（名义 GDP vs PPP）。

**验证标准**：数据来自 World Bank/IMF；口径差异明确标注；错误信息已标注为"不准确"；所有数据附带来源 URL 和年份。

## 场景 5: 历史小说背景研究（novel）

**背景**：一位历史小说作家正在创作一部以 19 世纪维多利亚时代伦敦为背景的小说。需要验证多个历史细节的准确性，包括人物名字的文化背景、当时使用的词汇词源、以及地理环境的准确性。

**操作流程（多节点联动）**：
1. **节点 A — 人名验证**：在 Behind the Name 上查询角色名"Eleanor"和"Reginald"的词源、流行年代和文化背景。确认这两个名字在 19 世纪英国确实流行。
2. **节点 B — 词汇词源验证**：在 Etymonline 上查询小说中使用的词汇（如"gentleman"、"parlor"、"gaslight"）的词源，确认它们在 19 世纪已存在。发现"gaslight"一词虽在 19 世纪已有煤气灯，但作为心理操控的含义（gaslighting）要到 20 世纪才出现。
3. **节点 C — 地理验证**：在 GeoNames 上查询小说中提到的伦敦地名（如Whitechapel、Mayfair），确认 19 世纪这些地名已存在。
4. **节点 D — 历史细节交叉验证**：在 Britannica 和 JSTOR 上查阅维多利亚时代的社会习俗、服饰、交通方式等历史背景，确保小说细节准确。

**验证标准**：人名词源来自 Behind the Name；词汇词源来自 Etymonline/OED；地名来自 GeoNames；历史背景有多源交叉验证；无时代错误（anachronism）。
