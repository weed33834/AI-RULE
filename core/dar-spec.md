# DAR (Domain Authority Registry) — 域权威注册表统一规范

> 本文件定义 DAR 的统一规范：来源分级、打分公式、路由规则、冲突策略、动态调整。
> 所有 Profile 的 DAR 配置（`capabilities/dar/dar-<domain>.yaml`）均遵循本规范。
> DAR 是深度搜索（`deep-search.md`）和真实性协议（`truth-protocol.md`）的底层支撑。

## §1 目的

传统搜索行为"全网乱搜→取前几条"存在三大问题：
1. **来源不可控**：SEO 垃圾、内容农场可能排在首位。
2. **权威性无法量化**：没有统一标准判断哪个来源更可信。
3. **领域知识缺失**：不同领域对时效性、共识性、权威性的要求截然不同。

DAR 通过预置权威源名录、打分公式、路由规则和领域知识，让搜索行为更有指向性。

## §2 模块结构

每个领域的 DAR 配置包含 5 个模块：

```yaml
source_registry:    # 权威源名录（T1-T4 分级）
scoring_protocol:   # 打分权重配置
routing_rules:      # 检索路由规则
domain_knowledge:    # 领域专属知识
conflict_policy:    # 冲突处理策略
```

## §3 Source Registry（来源分级）

### 3.1 T1-T4 四档统一分级

| Tier | 来源类型 | 可信度权重 | 使用方式 | 典型示例 |
|------|----------|------------|----------|----------|
| **T1** | 官方文档、同行评审论文、标准制定机构 | ×1.0 | 可直接引用 | Python docs、Nature、RFC、WHO |
| **T2** | 权威媒体、知名技术出版物、行业报告 | ×0.8 | 可引用，标注来源 | BBC、TechCrunch、Statista |
| **T3** | 技术博客、SO 高票回答、百科条目 | ×0.5 | 可引用，建议交叉验证 | Medium、Dev.to、Wikipedia |
| **T4** | 社区讨论、个人观点、社交媒体、匿名来源 | ×0.2 | 需交叉验证后才可引用 | Reddit、Twitter、匿名论坛 |

### 3.2 分级迁移映射

| 原分级 | DAR 分级 | 说明 |
|--------|----------|------|
| A / T-A / Level 1 | **T1** | 最高可信度 |
| B / T-B / Level 2 | **T2** | 高可信度 |
| C / T-C / Level 3 | **T3** | 中可信度 |
| D, E / T-D, T-E / Level 4, 5 | **T4** | 低可信度 |

### 3.3 来源纳入标准

- **T1 纳入标准**：同行评审期刊（SCI/SSCI）、官方标准文档、政府/国际组织发布、公认权威参考
- **T2 纳入标准**：知名媒体机构、行业领先出版物、经过验证的行业报告
- **T3 纳入标准**：高票社区回答（SO 100+ vote）、知名技术博客、百科条目
- **T4 纳入标准**：其余所有可访问来源，使用时必须交叉验证

## §4 Scoring Protocol（打分协议）

### 4.1 打分公式

```
Final Score = α × Relevance + β × Credibility + γ × Freshness + δ × Consensus
```

| 维度 | 含义 | 计算方式 |
|------|------|----------|
| **Relevance** | 与查询的相关性 | 0.0-1.0，基于关键词匹配和语义相关性 |
| **Credibility** | 来源可信度 | T1=1.0, T2=0.8, T3=0.5, T4=0.2 |
| **Freshness** | 信息时效性 | 见 §4.3 时效表 |
| **Consensus** | 多源共识度 | 1源=0.3, 2源=0.6, 3+源=1.0 |

### 4.2 默认权重

| 权重 | 默认值 | 说明 |
|------|--------|------|
| α (Relevance) | 0.40 | 相关性永远重要 |
| β (Credibility) | 0.30 | 可信度是核心 |
| γ (Freshness) | 0.15 | 时效性领域差异大 |
| δ (Consensus) | 0.15 | 共识度增强可信 |

权重必须满足：α + β + γ + δ = 1.0

### 4.3 时效表（Freshness Table）

| 信息类型 | 有效期 | 过期降权因子 | 超期处理 |
|----------|--------|-------------|----------|
| 安全漏洞/CVE | 3 个月 | ×0.3/月 | 必须查最新状态 |
| 市场数据/统计 | 6-12 个月 | ×0.5/年 | 标注日期，建议查最新 |
| 技术框架版本 | 1-2 年 | ×0.7/年 | 标注"可能已过时" |
| 实证研究 | 5-10 年 | ×0.8/5年 | 检查是否有新研究推翻 |
| 理论研究 | 10-20 年 | ×0.9/10年 | 经典理论不降权 |
| 定义/概念 | 长期有效 | ×1.0 | 通常无需更新 |
| 基础公理 | 永不过期 | ×1.0 | 无需时效检查 |

Freshness 值计算：`freshness = decay_factor ^ (age / period)`

### 4.4 共识度计算

| 独立来源数 | Consensus 值 | 说明 |
|-----------|-------------|------|
| 1 | 0.3 | 单一来源，可信度有限 |
| 2 | 0.6 | 两个独立来源，较可信 |
| 3+ | 1.0 | 多源共识，高度可信 |

"独立来源"定义：不同机构、不同作者、不同数据来源。同一机构的多篇文章不算独立。

## §5 Routing Rules（路由规则）

### 5.1 路由决策流程

```text
用户查询
  │
  ▼
匹配 DAR 路由规则（trigger 关键词）
  │
  ├─ 命中 → 查询 priority_sources（T1/T2 级源）
  │    │
  │    ├─ 找到 ≥2 个独立来源 → 输出（L0 完整验证）
  │    │
  │    ├─ 找到 1 个来源 → 查 fallback 源补充（L1 部分验证）
  │    │
  │    └─ priority_sources 无结果 → 全网搜索（L2 弱验证）
  │
  └─ 未命中 → 全网搜索（使用默认打分公式）
```

### 5.2 路由规则格式

```yaml
routing_rules:
  - trigger: ["关键词1", "关键词2"]    # 触发条件
    priority_sources: [T1源, T2源]      # 优先查询的源
    fallback: [T3源, T4源]              # 备选源
    min_tier: T2                        # 最低可信度门槛
    action: "must_verify"               # 附加动作
```

### 5.3 Fallback 链

| 级别 | 条件 | 行为 |
|------|------|------|
| L0 完整验证 | 2+ T1/T2 来源确认 | 正常输出，附来源 |
| L1 部分验证 | 1 个 T1/T2 或 2 个 T3 来源 | 输出但标注[中]置信度 |
| L2 弱验证 | 仅有 T3/T4 来源 | 输出但标注[低]置信度，建议用户验证 |
| L3 无法验证 | 无可靠来源 | 说"我无法确认这个信息"，不编造 |
| L4 矛盾 | 来源之间矛盾 | 呈现各方观点，不强行选边 |

## §6 Conflict Policy（冲突策略）

### 6.1 冲突优先级

```text
1. 同级别来源冲突 → 呈现各方观点，不强行选边
2. 不同级别来源冲突 → 优先高级别来源（T1 > T2 > T3 > T4）
3. 时效冲突 → 优先最新来源（除非旧来源是经典/公理）
4. 方法论差异导致冲突 → 说明差异原因
5. 用户指定来源 → 用户决策权最高，覆盖所有 DAR 规则
```

### 6.2 冲突处理矩阵

| 冲突类型 | 示例 | 处理方式 |
|----------|------|----------|
| 数据冲突 | A 说 5B，B 说 6B | 标注分歧，呈现范围 |
| 结论冲突 | A 说好，B 说不好 | 呈现双方论点 |
| 时效冲突 | A(2023) vs B(2024) | 优先最新 |
| 定义冲突 | 术语定义不同 | 呈现两种定义 |
| 级别冲突 | T1 vs T4 | 优先 T1 |

## §7 Domain Knowledge（领域知识）

每个领域配置包含：

| 字段 | 内容 | 作用 |
|------|------|------|
| `terminology` | 领域专业术语表 | 搜索时使用正确的专业关键词 |
| `conventions` | 领域写作/引用规范 | 确保输出符合领域惯例 |
| `common_pitfalls` | 常见误区和陷阱 | 避免常见错误 |
| `quality_signals` | 质量判断信号 | 快速判断来源质量 |

## §8 Dynamic Adjustment（动态调整）

### 8.1 场景化权重热调整

| 场景 | α | β | γ | δ | 调整说明 |
|------|-----|-----|-----|-----|----------|
| 默认 | 0.40 | 0.30 | 0.15 | 0.15 | 平衡型 |
| 安全漏洞查询 | 0.30 | 0.30 | 0.35 | 0.05 | 时效性最高 |
| 突发新闻 | 0.25 | 0.25 | 0.40 | 0.10 | 时效性主导 |
| 历史事实 | 0.35 | 0.20 | 0.05 | 0.40 | 共识度主导 |
| 学术引用 | 0.30 | 0.40 | 0.15 | 0.15 | 可信度主导 |
| 创意写作 | 0.35 | 0.20 | 0.05 | 0.40 | 共识度主导 |

### 8.2 用户覆盖层

用户决策权最高，可覆盖所有 DAR 规则：
- 指定来源 → 使用用户指定源，跳过路由
- 调整权重 → 用户指定权重覆盖默认/场景权重
- 禁用某源 → 从 source_registry 移除
- 禁用 DAR → 回退到全网搜索 + 默认打分
