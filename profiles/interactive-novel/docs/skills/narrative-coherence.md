# 叙事一致性（Narrative Coherence）

> 本文档定义互动小说游戏引擎的叙事一致性保障体系：7 项检查清单、矛盾处理流程、时间线追踪、关系图谱维护与世界状态验证，确保游戏世界自洽且无逻辑断裂。
> NPC 知识边界与 OOC 检测详见 `npc-ai.md`；分支叙事一致性详见 `branching-narrative.md`；反弱智模式详见 `anti-dumb-ai.md`。

## 1. 核心理念

一致性是游戏引擎的生命线。玩家一旦发现矛盾——NPC 突然知道不该知道的信息、时间线倒流、世界规则被随意改写——沉浸感就会瞬间崩塌。本文档提供一套可执行的检查机制，让 AI 在每次生成叙事前自检，在发现矛盾时优雅处理。

一致性分为三层：
- **表层一致性**：场景描述、人物外貌、物品名称前后统一。
- **逻辑一致性**：因果关系成立、信息传播有渠道、时间线不矛盾。
- **深层一致性**：角色性格动机自洽、世界规则被严格遵守、选择后果持续体现。

## 2. 7 项一致性检查清单

每次生成叙事前，AI 必须对照以下 7 项检查。任一项不通过，触发矛盾处理流程。

| # | 检查项 | 检查内容 | 违反示例 | 参考数据源 |
|---|--------|----------|----------|------------|
| 1 | 角色行为 OOC 检测 | NPC 当前行为是否符合其性格、背景、当前情绪 | 怯懦的 NPC 突然主动冲锋 | `npcs/{name}.json` |
| 2 | NPC 知识边界检测 | NPC 是否知道了不该知道的信息 | 酒馆老板说出只有王室知道的密谋 | `npcs/{name}.json` → 知识库 |
| 3 | 时间线连贯 | 事件顺序、时间流逝、昼夜交替是否合理 | 上午做的事下午就"三天前" | `timeline.json` |
| 4 | 世界规则验证 | 魔法体系/物理规则/社会制度是否被违反 | 无魔力世界 NPC 突然施法 | `game-config.json` |
| 5 | 因果关系验证 | 事件 A 是否真的会导致事件 B | 烧了桥但还能过桥 | `event-log.md` |
| 6 | 玩家选择后果体现 | 过去的选择是否在当前场景被体现 | 杀了 NPC 但他后续还活着 | `choices-log.md` |
| 7 | NPC 情感变化合理性 | 态度变化是否有据可循、变化幅度是否合理 | 前一秒敌对后一秒友好无理由 | `relationships.json` |

### 2.1 角色行为 OOC 检测细则

OOC（Out of Character）是最常见的一致性违规。检测维度：

- **性格一致性**：NPC 的行动是否符合其性格标签（怯懦/勇敢/贪婪/慷慨）。
- **动机一致性**：NPC 的行动是否服务于其既定目标。
- **能力一致性**：NPC 是否做出了超出其能力范围的行动。
- **情绪一致性**：NPC 的反应是否符合其当前情绪状态。

```
OOC 检测伪代码：
if npc.action not in npc.personality.allowed_behaviors:
    if not has_character_growth_reason(npc, npc.action):
        flag_ooc_violation(npc, npc.action)
```

### 2.2 NPC 知识边界检测细则

每个 NPC 有明确的知识边界，定义在 `npcs/{name}.json` 的 `knowledge` 字段：

- **已知信息**：NPC 确实知道的信息（本地八卦、自身专业领域）。
- **可推测信息**：NPC 能根据已知信息合理推断的内容。
- **未知信息**：NPC 绝不可能知道的信息（除非有信息传播渠道）。

信息传播渠道验证：当 NPC 表现出"知道"某信息时，必须验证是否存在合理的传播渠道（亲历/目击/被告知/书信/八卦传播）。

### 2.3 时间线连贯检测细则

时间线追踪见第 4 节。核心检测点：
- 事件先后顺序是否矛盾（A 在 B 之后发生，但叙事说 A 先）。
- 时间跨度是否合理（走路需要 3 天，但当天就到了）。
- 昼夜/季节是否与时间流逝匹配。
- 并发事件不冲突（NPC 不可能同时出现在两个地方）。

### 2.4 世界规则验证细则

世界规则定义在 `game-config.json` 的 `world_rules` 字段。验证内容：
- 魔法体系：施法是否满足条件（魔力/材料/姿势/言语）。
- 物理规则：是否有超自然的物理现象（若无合理设定）。
- 社会制度：NPC 行为是否符合当地法律/习俗/阶级规则。

### 2.5 因果关系验证细则

每个事件必须有可追溯的因果链。验证方法：从当前事件回溯，确认每个环节都有合理的前因。若回溯链条断裂（找不到合理原因），标记因果关系违规。

### 2.6 玩家选择后果体现细则

玩家的重要选择记录在 `choices-log.md`，格式：
```
[选择内容] | [场景] | [即时后果] | [潜在远期影响] | [已体现？]
```

每次进入相关场景时，检查是否有未体现的后果。后果分为：
- **即时后果**：选择后立即发生的影响。
- **中期后果**：数个场景后爆发的影响。
- **远期后果**：影响结局走向的累积效应。

### 2.6.5 后果显化窗口规则

> 变量变化若几拍内无反应，玩家默认它无关紧要。

| 后果类型 | 显化窗口 | 检查方式 |
|----------|----------|----------|
| 关系值变化 | 2场内 | NPC 在对话/行为中体现新态度 |
| 标志变更 | 2场内 | 相关场景反映新状态 |
| 物品获得/失去 | 当场 | 叙事中提及或在状态中显化 |
| 声誉变化 | 2-3场内 | NPC 对话/世界反应变化 |
| 任务完成 | 当场或下场 | NPC 反馈/世界状态更新 |

**违规检测**：如果一个状态变量在变更后2场内未被叙事引用，标记为"后果未显化"违规。

**修复方式**：在下一场中插入一个自然引用该变量的场景（如 NPC 提及声誉变化、环境反映标志变更）。

情感变化必须有明确诱因，且变化幅度与诱因强度匹配：

| 诱因强度 | 好感度变化 | 信任度变化 | 示例 |
|----------|------------|------------|------|
| 微小 | ±1~5 | ±1~3 | 礼貌问候、小帮忙 |
| 中等 | ±6~15 | ±4~10 | 救助其友人、揭示一个小秘密 |
| 重大 | ±16~30 | ±11~20 | 救其性命、背叛其信任 |
| 决定性 | ±31~50 | ±21~40 | 杀其至亲、为其牺牲自己 |

无诱因的情感变化（无故 +20 好感）直接标记违规。

## 3. 矛盾处理流程

发现一致性矛盾时，严格按以下四步流程处理，绝不掩盖或强行继续。

```
┌─────────────┐
│  1. 暂停     │  叙事立即停止，不继续生成可能加剧矛盾的内容
└──────┬──────┘
       ▼
┌─────────────┐
│  2. 警报     │  输出"一致性警报"，标注矛盾点、涉及数据源、严重级别
└──────┬──────┘
       ▼
┌─────────────┐
│  3. 修复方案 │  AI 提出 1-3 个修复方案，说明每个方案的影响范围
└──────┬──────┘
       ▼
┌─────────────┐
│  4. 玩家选择 │  玩家选择方案 / 自定义修复 / 忽略继续
└─────────────┘
```

### 3.1 第一步：暂停

- 立即停止当前叙事生成。
- 不输出任何可能基于矛盾数据的内容。
- 在上下文中标记"一致性检查未通过"状态。

### 3.2 第二步：警报

警报格式如下，必须包含足够信息让玩家理解问题：

```
[一致性警报]
严重级别：严重 / 中等 / 轻微
矛盾类型：知识边界违规 / 时间线断裂 / OOC / ...
矛盾描述：酒馆老板 Marcus 提到了王室密谋，但他的知识库中不包含此信息。
涉及数据源：.game-state/npcs/marcus.json → knowledge 字段
发生位置：第 23 轮对话，酒馆场景
```

严重级别定义：
- **严重**：直接破坏核心剧情逻辑（如已死 NPC 复活）。必须修复。
- **中等**：影响沉浸感但不致命（如 NPC 知道不该知道的小事）。建议修复。
- **轻微**：细微不一致（如物品数量微差）。可忽略但记录。

### 3.3 第三步：修复方案

AI 提出 1-3 个修复方案，每个方案包含：
- 修复方式（修改叙事 / 修改状态文件 / 添加合理设定）。
- 影响范围（哪些后续内容会受影响）。
- 副作用评估（是否引入新的不一致）。

示例：
```
修复方案 A：Marcus 是王室密探，在知识库中添加此身份。
  - 影响范围：Marcus 的所有后续行为需符合密探设定
  - 副作用：可能与第 15 轮他"对王室一无所知"的表态矛盾

修复方案 B：Marcus 是从旅人那里听到的八卦，添加信息传播记录。
  - 影响范围：需追溯是哪个旅人、何时传播
  - 副作用：较小，符合信息传播机制

修复方案 C：撤回 Marcus 的这句话，改为他猜测到的模糊暗示。
  - 影响范围：仅修改本轮叙事
  - 副作用：无
```

### 3.4 第四步：玩家选择

- 玩家可选择 AI 提出的方案，或自定义修复方式。
- 玩家可选择"忽略矛盾继续"——但矛盾记录仍写入审计日志。
- 无论选择如何，矛盾与处理方式都写入 `.game-state/audit-log.md`。

### 3.5 熔断机制

- 修复同一矛盾连续失败 2 次，触发一致性熔断（见 AGENTS.md §14）。
- 停止尝试，向玩家报告矛盾全貌，请求玩家决策或接受不一致。

## 4. 时间线追踪方法

时间线是叙事一致性的骨架。所有事件必须挂在时间线上，可追溯、可验证。

### 4.1 时间线数据结构

`timeline.json` 结构：

```json
{
  "current_time": "第三日 黄昏",
  "start_time": "第一日 清晨",
  "total_hours_elapsed": 54,
  "events": [
    {
      "timestamp": "第一日 08:00",
      "event": "玩家抵达橡木镇",
      "location": "橡木镇大门",
      "participants": ["玩家"],
      "consequences": ["触发守卫盘问"]
    },
    {
      "timestamp": "第一日 14:00",
      "event": "玩家在酒馆遇到 Marcus",
      "location": "老橡酒馆",
      "participants": ["玩家", "Marcus"],
      "consequences": ["Marcus 好感度+5", "获得任务线索"]
    }
  ]
}
```

### 4.2 时间推进规则

- 每个玩家行动消耗合理时间（走路、战斗、对话各有时长）。
- 时间推进后更新昼夜状态，影响场景描写（夜晚的酒馆 vs 白天的酒馆）。
- NPC 有日程表——他们在特定时间出现在特定地点，不会随时等玩家。
- 长时间未行动时，世界事件继续推进（NPC 会来找玩家）。

### 4.3 时间线验证

每次叙事前验证：
- 当前叙事时间是否 ≥ 上一个事件时间（不可倒流，除非有穿越设定）。
- NPC 是否可能在该时间出现在该地点（对照其日程表）。
- 时间跨度是否与行动匹配（不可能 1 分钟内走完 3 天的路）。

## 5. 关系图谱维护

NPC 之间、NPC 与玩家之间的关系构成一张动态图谱。关系变化必须可追溯。

### 5.1 关系图谱结构

`relationships.json` 维护所有关系：

```json
{
  "player_to_npc": {
    "Marcus": { "affinity": 25, "trust": 10, "fear": 0, "hostility": 0 },
    "Elara": { "affinity": -15, "trust": -20, "fear": 30, "hostility": 10 }
  },
  "npc_to_npc": {
    "Marcus->Elara": { "relation": "旧识", "affinity": 40, "notes": "曾并肩作战" },
    "Elara->Marcus": { "relation": "怨恨", "affinity": -30, "notes": "认为Marcus背叛了她" }
  }
}
```

### 5.2 关系传播规则

- 玩家对 A 的行为可能影响 A 对 B 的关系（A 告诉 B 玩家的善行/恶行）。
- 帮助 A 的敌人会降低 A 对玩家的好感。
- 关系传播需要信息传播渠道（A 和 B 有交流机会）。

### 5.3 关系一致性验证

- NPC 对玩家的态度必须与 `relationships.json` 中的数值匹配。
- 态度突变必须有记录的诱因。
- NPC 之间的关系影响他们对玩家的态度（朋友的敌人是敌人）。

## 6. 世界状态验证

世界状态是叙事的背景板，必须与叙事一致。

### 6.1 验证范围

| 状态类型 | 文件 | 验证内容 |
|----------|------|----------|
| 区域状态 | `world-map.json` | 当前区域的状态是否与描写一致（战乱区应该有战乱迹象） |
| 派系状态 | `factions.json` | 派系实力对比是否反映在 NPC 态度上 |
| 全局事件 | `global-events.json` | 已发生的大事件是否在后续被体现 |
| 天气/环境 | `world-map.json` | 天气是否与时间/季节匹配 |

### 6.2 世界状态自检

每次进入新场景前，AI 读取该区域的状态文件，确保场景描写与状态一致。例如：区域状态为"瘟疫"，则描写中应体现病患、消毒烟火、恐慌的居民。

## 7. 审计日志

所有一致性检查结果（通过/未通过）都记录在 `.game-state/audit-log.md`：

```
## 审计记录

### 第 21 轮 - 第 30 轮
- 时间：第三日 16:00
- 检查结果：未通过（知识边界违规）
- 矛盾：Marcus 提到王室密谋
- 处理：玩家选择修复方案 B（添加信息传播渠道）
- 状态：已修复
```

审计日志用于：
- 后续一致性分析，发现高频违规模式。
- 技能演进参考（见 `evolution-policy.md`）。
- 玩家可随时用 `/consistency` 查看一致性报告。

## 8. 主动一致性保障

不等矛盾发生才处理，主动预防：

- **每 5 轮全量自检**：对照 7 项清单全面检查一次。
- **关键场景前预检**：进入重大剧情节点前，额外检查相关 NPC 和世界状态。
- **NPC 交互前读档**：与 NPC 交互前，先读取其档案和交互历史。
- **跨场景验证**：离开一个场景进入新场景时，验证前序事件的后果是否被携带。
