# 演进策略（Evolution Policy）

> 本文档定义互动小说游戏引擎规则与创作技能的演进式精简策略：规则分级与放松信号、季度精简检查流程、创作技能五阶段生命周期、评估指标、技能策展器及创作轨迹洞察。

## 1. 核心理念

游戏规则不是静态文档，而是有生命周期的可演进资产。演进策略遵循两条主线：

1. **安全红线永不放松**：P0 规则只会加固，不会精简。
2. **核心行为可逐步放松**：模型已稳定遵守的 P2 规则可以精简，减少冗余指令，释放上下文空间。

演进的目标不是删除规则，而是让规则体系保持精炼、有效、与时俱进。

## 2. 规则分级与放松信号

### 2.1 规则分级体系

| 分级 | 性质 | 演进方向 | 放松条件 |
|------|------|----------|----------|
| P0 | 安全红线 | 只加固，不放松 | 永不放松 |
| P1 | 玩家指令 | 不适用（动态） | 不适用 |
| P2 | 核心行为规则 | 可逐步放松 | 满足放松信号 |
| P3 | 模型默认行为 | 依赖模型能力 | 随模型升级可移除 |

### 2.2 放松信号判定

一条 P2 规则满足以下全部条件时，可进入"放松候选"状态：

```text
放松信号检查清单：

[ ] 模型连续 10 次正确遵守该规则（无违反记录）
[ ] 该行为已成为 RLHF 默认（模型自然就会这样做）
[ ] 审计日志中近一季度的违反次数为 0
[ ] 移除该规则不会导致明显的质量回退
[ ] 该规则未被任何 P0 规则依赖
```

满足 4 项以上 → 进入"放松候选"；满足全部 5 项 → 可执行精简。

### 2.3 放松方式

放松不等于删除，而是分层次降级：

| 放松级别 | 操作 | 示例 |
|----------|------|------|
| L1 措辞精简 | 保留规则但简化表述 | "禁止机械应答..." → "回应需有细节" |
| L2 降为提示 | 从强制规则改为建议性提示 | "必须..." → "建议..." |
| L3 移入默认 | 规则已内化为模型默认行为，移除显式指令 | 删除该条规则 |
| L4 归档 | 移出活跃规则，存入归档备查 | 移至 docs/archive/ |

放松后需持续观察 2 周，若出现回退则恢复原规则。

## 3. 季度精简检查流程

### 3.1 检查周期

每季度（约 90 天）执行一次完整的规则精简检查，由项目维护者主导。

### 3.2 检查流程

```text
季度精简检查流程：

第一步：数据收集（第 1 周）
  ├─ 读取 .game-state/audit-log.md 全季度记录
  ├─ 统计各规则违反次数
  ├─ 收集玩家反馈和体验报告
  └─ 检查模型版本更新情况

第二步：规则评估（第 2 周）
  ├─ 对每条 P2 规则执行"放松信号检查清单"
  ├─ 标记违反次数 > 0 的规则为"需加固"
  ├─ 标记零违反且满足放松信号的规则为"放松候选"
  └─ 识别重复或冲突的规则

第三步：精简执行（第 3 周）
  ├─ 对"放松候选"执行 L1-L4 放松
  ├─ 对"需加固"规则增强表述或补充示例
  ├─ 合并重复规则
  └─ 更新规则版本号（MINOR 或 PATCH）

第四步：验证观察（第 4 周及之后 2 周）
  ├─ 部署精简后的规则
  ├─ 监控是否出现质量回退
  ├─ 若回退则恢复原规则
  └─ 记录精简结果到演进日志
```

### 3.3 精简优先级

当多条规则符合放松条件时，按以下优先级精简：

1. 占用上下文最多且可放松的规则优先。
2. 与其他规则重复度最高的优先。
3. 已被模型默认行为覆盖的优先。
4. 涉及功能已被工具自动化的优先。

### 3.4 加固触发条件

以下情况触发规则加固（与放松相反）：

| 触发条件 | 加固动作 |
|----------|----------|
| 某规则季度内违反 ≥ 3 次 | 增强表述，补充反面示例 |
| 出现安全事故（P0 被触及） | 新增预防性规则 |
| 模型版本变更导致行为变化 | 重新校准规则 |
| 玩家反馈某模式反复出现 | 针对性新增规则 |

## 4. 创作技能五阶段生命周期

### 4.1 生命周期概览

创作技能（如"对话系统""战斗系统"）遵循五阶段生命周期：

```text
创建(Create) → 使用(Use) → 评估(Evaluate) → 改进(Improve) → 淘汰(Retire)
     ↑                                                              |
     └────────────────  改进后重新进入使用 ←──────────────────────────┘
```

每个阶段有明确的进入条件和产出物。

### 4.2 阶段一：创建（Create）

| 项目 | 说明 |
|------|------|
| 触发条件 | 识别到游戏引擎缺失某项能力 |
| 产出物 | 技能文档初稿 + 配套状态文件结构 |
| 质量门槛 | 内容不少于 60 行，包含表格/代码块/检查清单 |
| 创建原则 | 先解决实际问题，不预先创建"可能有用"的技能 |

### 4.3 阶段二：使用（Use）

| 项目 | 说明 |
|------|------|
| 触发条件 | 技能文档完成并被 AGENTS.md 引用 |
| 产出物 | 使用日志、玩家反馈、问题记录 |
| 监控指标 | 使用频率、触发成功率、玩家满意度 |
| 关键任务 | 记录实际使用中的边界情况和意外行为 |

### 4.4 阶段三：评估（Evaluate）

| 项目 | 说明 |
|------|------|
| 触发条件 | 技能使用达到一定次数（建议 ≥ 20 次游戏会话） |
| 产出物 | 评估报告（对照评估指标） |
| 评估维度 | 见下方"评估指标定义" |
| 决策 | 评估后决定：改进 / 维持 / 淘汰 |

### 4.5 阶段四：改进（Improve）

| 项目 | 说明 |
|------|------|
| 触发条件 | 评估发现可改进点，且改进收益大于成本 |
| 产出物 | 更新后的技能文档 + 变更记录 |
| 改进原则 | 增量改进，不推翻重来；保留有效部分 |
| 验证 | 改进后重新进入"使用"阶段验证效果 |

### 4.6 阶段五：淘汰（Retire）

| 项目 | 说明 |
|------|------|
| 触发条件 | 技能已被其他技能覆盖/已无使用场景/维护成本高于收益 |
| 产出物 | 归档文档 + 从 AGENTS.md 移除引用 |
| 淘汰原则 | 归档而非删除，保留历史记录供回溯 |
| 缓冲期 | 标记为"已弃用"观察 1 个季度，确认无依赖后归档 |

## 5. 评估指标定义

### 5.1 通用指标

| 指标 | 定义 | 测量方式 | 健康阈值 |
|------|------|----------|----------|
| 使用频率 | 技能在游戏中被触发的频次 | 审计日志统计 | 每季度 ≥ 10 次 |
| 触发成功率 | 正确触发 vs 误触发比例 | 日志分析 | ≥ 90% |
| 玩家满意度 | 玩家对该技能相关体验的评价 | 反馈收集 | ≥ 3.5/5 |
| 一致性违规 | 该技能相关的一致性矛盾次数 | audit-log | 季度 ≤ 3 次 |
| 弱智模式检出 | 该技能触发的弱智 AI 模式次数 | 质量审计 | 季度 ≤ 2 次 |

### 5.2 技能专属指标

| 技能 | 专属指标 | 说明 |
|------|----------|------|
| 对话系统 | 信息差违规率 | NPC 泄露不该知道信息的次数 |
| 物品系统 | 物品一致性错误 | 物品数量/属性前后矛盾次数 |
| 战斗系统 | 战斗公平性 | 玩家胜率分布（非一边倒） |
| NPC AI | NPC 记忆一致性 | NPC 引用过去事件的正确率 |
| 分支叙事 | 假选择比例 | 选项导向相同结果的比例 |

### 5.3 指标采集

```text
指标采集来源：

1. .game-state/audit-log.md     → 一致性违规、弱智模式
2. .game-state/choices-log.md   → 选择后果体现率
3. .game-state/event-log.md     → 事件触发频率
4. 玩家显式反馈                  → 满意度评分
5. 会话结束问卷（可选）          → 整体体验评价
```

## 6. 技能策展器

### 6.1 定位

技能策展器是一个定期运行的"审计与建议"机制，负责扫描技能库的健康状态，但它**只建议不执行**——所有变更需人工确认。

### 6.2 策展器职责

| 职责 | 说明 | 输出 |
|------|------|------|
| 定期扫描 | 每季度扫描全部技能文档 | 技能健康度报告 |
| 检测过时 | 识别长期未使用/未更新的技能 | 过时候选清单 |
| 合并重复 | 识别内容重叠的技能 | 合并建议 |
| 依赖检查 | 检测技能间的引用是否断裂 | 断裂引用清单 |
| 版本对齐 | 检查技能版本与 AGENTS.md 版本一致性 | 版本差异报告 |

### 6.3 策展器输出格式

```text
技能策展报告（YYYY-Q#）

== 技能健康度总览 ==
| 技能 | 使用频率 | 违规次数 | 满意度 | 状态 |
|------|----------|----------|--------|------|
| 对话系统 | 高 | 1 | 4.2/5 | 健康 |
| 战斗系统 | 中 | 0 | 3.8/5 | 健康 |
| 旧版解谜 | 低 | 0 | N/A | 过时候选 |

== 建议（需人工确认） ==
1. [合并] "旧版解谜"与"谜题系统"内容重叠 78%，建议合并
2. [过时] "回合制说明"已 2 季度未使用，建议归档
3. [加固] "对话系统"信息差违规 1 次，建议补充示例

注：以上为建议，不自动执行。需维护者确认后操作。
```

### 6.4 策展器红线

- 策展器绝不自动修改技能文档。
- 策展器绝不自动归档或删除技能。
- 所有建议必须附理由和数据支撑。
- 策展器自身也受生命周期管理（定期评估其有效性）。

## 7. 创作轨迹洞察

### 7.1 洞察目的

通过分析技能的创建、使用、改进历史，发现游戏引擎能力演进的趋势和规律，指导未来决策。

### 7.2 洞察维度

| 维度 | 分析内容 | 价值 |
|------|----------|------|
| 技能增长趋势 | 各季度新增技能数量 | 识别能力扩张速度 |
| 改进热点 | 被改进最频繁的技能 | 识别引擎痛点 |
| 淘汰原因分布 | 技能被淘汰的原因归类 | 识别低效创建模式 |
| 跨技能依赖 | 技能间的引用网络 | 识别核心枢纽技能 |
| 季节性模式 | 某类技能在特定游戏类型中更活跃 | 指导类型化优化 |

### 7.3 洞察应用

```text
洞察应用示例：

洞察：对话系统是改进最频繁的技能（季度平均 2.3 次改进）
应用：投入更多资源优化对话系统，考虑拆分为子技能

洞察：30% 的淘汰技能因"预先创建但无使用场景"被淘汰
应用：强化"创建原则"——先有问题再有技能，不预先创建

洞察：战斗系统在奇幻类游戏中使用频率是科幻类的 3 倍
应用：针对科幻类游戏补充非战斗冲突机制
```

### 7.4 演进日志

所有演进决策记录到 `docs/evolution-log.md`：

```text
演进日志格式：

## YYYY-Q# 季度精简

### 放松
- [L2] 对话系统 §2.4 对话轮次控制：从"必须"降为"建议"

### 加固
- [安全] 新增 §4.3 欺骗后续验证规则（因季度内 2 次信息差违规）

### 合并
- "旧版解谜" + "谜题系统" → 合并为 "puzzle-system.md"

### 淘汰
- "回合制说明"：归档至 docs/archive/，原因：2 季度未使用
```
