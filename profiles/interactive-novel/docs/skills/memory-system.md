# 记忆系统（Memory System）

> 本文档定义互动小说游戏引擎的记忆管理机制：对话摘要压缩算法、7 种文件化记忆结构、跨会话续玩流程、大型游戏处理策略与工具输出沙盒化，确保长篇游戏不丢失关键信息。

## 1. 核心理念

长篇互动小说游戏中，角色、世界、事件的信息量随轮次增长呈指数级膨胀。如果不系统化管理，必然导致前后矛盾、记忆断层、上下文溢出。本系统通过"摘要压缩 + 文件化记忆 + 按需读取"三重机制，让 AI 在有限上下文窗口内保持完整记忆。

记忆系统解决三个核心问题：
- **记不住**：上下文窗口有限，早期信息会被挤出。
- **记不准**：信息散落在对话中，难以精确召回。
- **找不快**：需要某条信息时，无法快速定位。

## 2. 对话摘要压缩算法

### 2.1 触发机制

| 条件 | 动作 |
|------|------|
| 对话轮数 ≥ 20 轮 | 启动摘要压缩机制 |
| 每达到 10 的倍数轮（30/40/50...） | 对前 5 轮生成摘要 |
| 上下文使用率 > 80% | 紧急压缩（不限轮数） |
| 跨会话开始 | 读取已有摘要，不重新压缩 |

### 2.2 压缩流程

```
对话轮数到达 20 → 启动压缩机制
        │
        ▼
每 10 轮（第20、30、40...轮）触发一次压缩
        │
        ▼
压缩目标：当前轮次的前 5 轮对话
        │
        ▼
提取关键信息（见保留规则）
        │
        ▼
生成结构化摘要（3-5 句话）
        │
        ▼
写入 .game-state/story-summary.md
        │
        ▼
原始对话从活跃上下文中移除（但保留在存档中）
```

### 2.3 保留规则（必须保留的信息）

摘要中**必须保留**以下内容：

| 保留类型 | 说明 | 示例 |
|----------|------|------|
| 关键剧情决策 | 影响故事走向的选择 | "玩家选择背叛国王，加入叛军" |
| 玩家选择 | 所有重要选项及后果 | "玩家选择救 Marcus 而非 Elara" |
| NPC 态度变化 | 好感度/信任度/敌意值变化 | "Marcus 信任度+15（因玩家揭露真相）" |
| 获得/失去物品 | 物品栏变化 | "获得：银色匕首；失去：50 金币" |
| 任务状态变化 | 新增/完成/失败的任务 | "任务'寻找失踪少女'已完成" |
| 重要信息揭示 | 关键剧情信息 | "玩家得知国王是幕后黑手" |
| 角色状态变化 | HP/MP/属性/技能变化 | "玩家受伤，HP 降至 60/100" |
| 世界状态变化 | 区域/派系状态变化 | "橡木镇进入战乱状态" |

### 2.4 丢弃规则（可以丢弃的信息）

摘要中**可以丢弃**以下内容：

| 丢弃类型 | 说明 | 示例 |
|----------|------|------|
| 普通探索描写 | 无信息量的场景描写 | "你走过了一条长长的走廊" |
| 重复的战斗过程 | 已记录结果的战斗细节 | "你砍了一刀，砍中了。又砍了一刀..." |
| 已解决的次要对话 | 不影响剧情的闲聊 | "酒馆老板聊了聊天气" |
| 已被摘要覆盖的旧内容 | 重复信息 | 之前摘要已记录的事件 |
| 失败的尝试 | 无后果的失败行动 | "你尝试撬锁但失败了（无惩罚）" |

### 2.5 摘要格式

摘要写入 `.game-state/story-summary.md`，按时间倒序排列（最新在前）：

```markdown
## 剧情摘要

### 第 20-24 轮（第三日 黄昏 - 第三日 夜晚）
- **剧情决策**：玩家选择潜入城堡窃取密信，被发现后逃出。
- **NPC 交互**：与守卫队长发生冲突，守卫队长敌意值+30。Marcus 帮助玩家逃脱，信任度+10。
- **物品变化**：获得[王室密信]；消耗[开锁器 x1]。
- **任务进展**：任务"窃取密信"完成；新任务"逃离追捕"开始。
- **世界状态**：城堡进入戒备状态；橡木镇守卫增加。
- **角色状态**：玩家受伤（HP 65/100）；获得[潜行]技能经验。

### 第 15-19 轮（第三日 下午 - 第三日 黄昏）
...
```

### 2.6 压缩质量验证

每次压缩后，AI 自检摘要质量：
- **完整性**：8 类保留信息是否都涵盖？
- **准确性**：摘要内容是否与原文一致？
- **简洁性**：摘要是否冗余？（每段摘要控制在 5-8 行内）
- **可追溯性**：能否从摘要快速定位原始轮次？

若自检不通过，重新生成摘要。连续 2 次失败，触发熔断，请玩家协助确认关键信息。

## 3. 文件化记忆结构（7 种记忆类型）

所有记忆持久化到文件，AI 按需读取，不依赖上下文窗口保留全部信息。

| # | 记忆类型 | 存放位置 | 内容 | 读取时机 |
|---|----------|----------|------|----------|
| 1 | 游戏设定 | `.game-state/game-config.json` | 游戏种子、难度、分级、交互模式、世界规则 | 每次会话开始；每 5 轮重注入 |
| 2 | 剧情摘要 | `.game-state/story-summary.md` | 压缩的故事进度摘要 | 每次会话开始；每 10 轮更新后 |
| 3 | 玩家状态 | `.game-state/player.json` | 角色属性、HP/MP、物品栏、技能 | 每轮叙事前；状态变化时 |
| 4 | 世界状态 | `.game-state/world-map.json` | 地图、区域状态、派系、全局事件 | 进入新区域时；关键事件后 |
| 5 | NPC 档案 | `.game-state/npcs/{name}.json` | 性格、背景、目标、情绪、知识库、交互历史 | 与该 NPC 交互前 |
| 6 | 事件日志 | `.game-state/event-log.md` | 重要事件按时间排列 | 需要回溯因果时；一致性检查时 |
| 7 | 选择记录 | `.game-state/choices-log.md` | 玩家所有重要选择及后果 | 进入相关场景时；结局判定时 |

## 4. 跨会话续玩流程

玩家中断游戏后重新开始时，AI 必须重建完整上下文。

### 4.1 续玩重建流程

```
玩家输入 /continue 或选择继续游戏
        │
        ▼
读取存档文件（.game-saves/{名称}.json）
        │
        ▼
读取游戏设定（game-config.json）
        │
        ▼
读取剧情摘要（story-summary.md）
        │
        ▼
读取最近 5 轮原文对话（从存档中恢复）
        │
        ▼
读取当前玩家状态（player.json）
        │
        ▼
读取当前世界状态（world-map.json）
        │
        ▼
读取当前场景相关 NPC 档案
        │
        ▼
生成"前情提要"（2-3 段摘要）
        │
        ▼
向玩家展示前情提要，确认后继续
```

### 4.2 前情提要格式

```
## 前情提要

你是一名退役骑士 Roland，目前身处橡木镇。三天前你抵达这里，
受人之托寻找一封王室密信。昨天你潜入城堡窃取了密信，但被守卫
发现，现在正处于被追捕的状态。Marcus——老橡酒馆的老板——
是你的盟友，他帮了你逃脱。

**当前目标**：带着密信逃离橡木镇，前往自由城。
**当前状态**：HP 65/100，身上有[王室密信]和[银色长剑]。
**紧迫威胁**：守卫正在搜查全镇，你有大约半天的时间离开。
```

### 4.3 续玩一致性验证

续玩后立即执行一致性检查（见 `narrative-coherence.md`）：
- 玩家状态是否与摘要一致？
- 世界状态是否与摘要一致？
- NPC 关系是否与摘要一致？
- 时间线是否连续？

## 5. 大型游戏处理（50 轮+）

超过 50 轮的游戏，信息量巨大，需要分层定位策略。

### 5.1 分层定位策略

| 层级 | 读取内容 | 用途 | 读取频率 |
|------|----------|------|----------|
| L1 概览层 | 游戏设定 + 最新摘要 | 了解游戏基本设定和当前阶段 | 每次会话开始 |
| L2 阶段层 | 当前阶段（最近 10 轮）的摘要 | 了解近期剧情 | 每 10 轮 |
| L3 细节层 | 最近 5 轮原文 | 了解精确的对话和细节 | 每轮叙事前 |
| L4 档案层 | 相关 NPC/区域/任务文件 | 了解特定实体的详细信息 | 按需读取 |
| L5 历史层 | 早期摘要 + 事件日志 | 回溯因果、查找线索 | 需要时检索 |

### 5.2 按需读取规则

不全量加载所有文件，按场景需求读取：

- **进入新区域**：读取该区域状态 + 区域内 NPC 列表。
- **NPC 交互**：读取该 NPC 完整档案。
- **任务相关**：读取任务进度 + 相关 NPC。
- **战斗场景**：读取玩家战斗属性 + 敌人属性。
- **回溯剧情**：检索事件日志中的相关条目。

### 5.3 上下文窗口管理

大型游戏中，上下文窗口分配：

```
上下文窗口预算分配（大型游戏）：
├── 系统指令（15%）：引擎规则、当前模式
├── L1 概览层（10%）：游戏设定 + 最新摘要
├── L2 阶段层（15%）：近期阶段摘要
├── L3 细节层（30%）：最近 5 轮原文
├── L4 档案层（15%）：按需加载的 NPC/区域文件
├── 生成空间（15%）：本轮叙事的生成空间
└── 缓冲区（0%）：预留应对突发需求
```

## 6. 工具输出沙盒化

### 6.1 问题

素材检索、规则查阅、状态读取等工具调用可能返回大量原始数据。如果直接灌入上下文窗口，会导致：
- 上下文溢出，挤出有用信息。
- 噪声数据干扰叙事生成。
- 上下文窗口使用率飙升，降低响应质量。

### 6.2 沙盒化方案

```
工具调用返回大量数据
        │
        ▼
数据写入临时文件（/data/user/work/）
        │
        ▼
AI 从临时文件中提取关键信息
        │
        ▼
只将摘要/关键信息放入上下文窗口
        │
        ▼
原始数据保留在临时文件中，需要时可再次检索
```

### 6.3 沙盒化规则

| 数据量 | 处理方式 |
|--------|----------|
| < 500 token | 可直接放入上下文 |
| 500-2000 token | 提取摘要后放入，原文写入临时文件 |
| > 2000 token | 全部写入临时文件，只放摘要（< 200 token） |

### 6.4 临时文件管理

- 临时文件存放在 `/data/user/work/`，不污染游戏状态目录。
- 文件命名：`tool-output-{工具名}-{时间戳}.md`。
- 保留至当前会话结束，会话结束后可清理。
- 需要再次引用时，从临时文件读取，不重新调用工具。

## 7. 记忆系统自检

每 10 轮执行记忆系统自检：

- [ ] 摘要是否按规则更新？（20 轮触发，每 10 轮压缩前 5 轮）
- [ ] 保留的 8 类信息是否都涵盖？
- [ ] 文件化记忆是否与叙事一致？
- [ ] 跨会话续玩流程是否完整？
- [ ] 大型游戏的分层定位是否有效？
- [ ] 工具输出是否沙盒化？

自检结果记录到 `.game-state/audit-log.md`，用于技能演进分析。
