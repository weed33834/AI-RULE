# 结局系统 (Ending System)

> 本文档定义互动小说游戏的结局判定与生成体系：5 种结局类型、结局判定算法、结局生成流程、尾声设计。
> 结局是玩家所有选择的终局——好的结局让玩家感到"这是我的故事"。
> 与 `branching-narrative.md` 互补——分支叙事定义"故事怎么分叉"，结局系统定义"故事怎么收束"。
> 与 `game-evaluation.md` 互补——评估度量整体体验，结局是体验的最后印象。

## §1 设计原则

- **选择有重量**：结局必须体现玩家的关键选择，不能所有路线汇归同一结局。
- **意料之外情理之中**：好结局让玩家惊讶但回头看觉得合理。
- **情感闭环**：结局要回收故事开头的情感承诺。
- **留有余韵**：不要把所有事情都解释清楚，留一些想象空间。
- **拒绝机械**：不能是"你的分数 > 80 所以是好结局"式的机械判定。

## §2 五种结局类型

| 类型 | 定义 | 触发条件 | 适用类型 |
|------|------|----------|----------|
| 胜利结局 | 玩家达成了核心目标 | 主线目标完成 + 关键选择正确 | 冒险、英雄故事 |
| 悲剧结局 | 玩家失败了，但有尊严 | 主线目标失败 + 角色弧光完整 | 悲剧、战争 |
| 反转结局 | 真相颠覆了玩家认知 | 关键信息在后期揭露 | 悬疑、恐怖 |
| 开放结局 | 故事未完全收束，留有想象 | 主线部分完成 + 玩家选择"离开" | 文学向、系列作 |
| 隐藏结局 | 需要特殊条件触发 | 满足隐蔽的特定条件组合 | 所有类型（奖励性） |

### 2.1 结局类型组合

一个游戏可以有多个结局，玩家到达哪个取决于选择路径：

```
结局分布示例（3-7 个结局）：

结局 A (胜利·完美) ← 极少数玩家能达到，需要全程高质量选择
结局 B (胜利·代价) ← 主流胜利路线，有牺牲
结局 C (悲剧·壮烈) ← 失败但角色弧光完整
结局 D (悲剧·徒劳) ← 失败且角色弧光未完成
结局 E (开放·离别) ← 玩家选择中途退出主线
结局 F (隐藏·真相) ← 满足特定条件（如收集了所有线索）
```

## §3 结局判定算法

### 3.1 多维结局评分

结局不是单一变量决定的，而是多维评分的综合结果：

```json
{
  "ending_factors": {
    "main_quest_progress": 0.85,     // 主线完成度 (0-1)
    "key_choices_alignment": 0.60,    // 关键选择与核心目标的契合度 (0-1)
    "character_arc_completion": 0.90, // 角色弧光完成度 (0-1)
    "relationship_state": {           // 关键关系最终状态
      "mentor": "alive_and_trusting",
      "rival": "defeated_or_reconciled",
      "love_interest": "alive_but_distant"
    },
    "moral_alignment": "pragmatic",   // 道德倾向 (heroic/pragmatic/anti-hero)
    "world_state": "stable"           // 世界最终状态 (stable/unstable/collapsing)
  }
}
```

### 3.2 判定流程

```
┌─────────────────────────────────┐
│ 1. 检查硬性触发条件              │
│    (如: 主角死亡 → 悲剧结局)     │
└──────────┬──────────────────────┘
           ▼ (无硬性触发)
┌─────────────────────────────────┐
│ 2. 计算主线完成度                │
│    > 0.7 → 胜利路线              │
│    0.3-0.7 → 开放路线            │
│    < 0.3 → 悲剧路线              │
└──────────┬──────────────────────┘
           ▼
┌─────────────────────────────────┐
│ 3. 检查隐藏结局条件              │
│    (如: 收集了所有线索 → 隐藏结局) │
└──────────┬──────────────────────┘
           ▼ (无隐藏触发)
┌─────────────────────────────────┐
│ 4. 细分结局类型                  │
│    根据角色弧光/关系/道德倾向判定 │
└──────────┬──────────────────────┘
           ▼
┌─────────────────────────────────┐
│ 5. 生成结局叙事                  │
│    回收伏笔 + 情感闭环 + 尾声    │
└─────────────────────────────────┘
```

### 3.3 关键选择权重

不同选择对结局的影响权重不同：

| 选择类型 | 权重 | 示例 |
|----------|------|------|
| 决定性选择 | 极高 | 是否杀死关键 NPC |
| 方向性选择 | 高 | 选择信任还是怀疑 |
| 积累性选择 | 中 | 多次帮助/伤害同一 NPC |
| 情境性选择 | 低 | 战斗中的战术选择 |
| 装饰性选择 | 无 | 选择穿什么颜色的衣服 |

## §4 结局生成流程

### 4.1 回收清单

结局生成前，检查所有需要回收的元素：

| 回收类型 | 检查内容 | 数据源 |
|----------|----------|--------|
| 伏笔回收 | 所有埋设的伏笔是否在结局中体现 | `choices-log.md` |
| 角色弧光 | 主要角色是否有始有终 | `npcs/*.json` |
| 关系闭环 | 关键关系是否有交代 | `relationships.json` |
| 世界状态 | 玩家的行为如何改变了世界 | `world-map.json` |
| 主题呼应 | 结局是否呼应了故事开头的主题 | `game-config.json` |

### 4.2 结局叙事结构

```
结局叙事 = 回顾 + 高潮 + 收束 + 尾声

回顾 (200-400字):
  → 用几句话回顾玩家的旅程
  → 提及最关键的 2-3 个选择
  → 让玩家感到"这是我的故事"

高潮 (500-1000字):
  → 最终场景的呈现
  → 根据结局类型决定情感基调
  → 确保行动感，不要全是旁白

收束 (300-500字):
  → 主线问题的最终解答
  → 角色的最终状态交代
  → 世界的变化体现

尾声 (100-300字):
  → 时间跳跃后的一个画面
  → 留白，不解释所有事情
  → 情感余韵
```

### 4.3 结局质量检查

| # | 检查项 | 标准 |
|---|--------|------|
| 1 | 选择体现 | 至少 3 个玩家的关键选择在结局中有体现 |
| 2 | 伏笔回收 | 80%+ 的伏笔在结局中回收或暗示 |
| 3 | 情感闭环 | 开头的情感承诺在结尾得到回应 |
| 4 | 角色交代 | 所有主要角色有最终状态 |
| 5 | 合理性 | 回头看，结局是玩家选择的自然结果 |
| 6 | 独特性 | 与其他结局有明显区别，不是换皮 |
| 7 | 余韵 | 结局后玩家有"想再玩一次"或"想回味"的感觉 |

## §5 尾声设计

尾声是结局之后的一小段文字，给予情感收束：

### 5.1 尾声类型

| 类型 | 特征 | 示例 |
|------|------|------|
| 时间跳跃 | 跳到数月/数年后的一个瞬间 | "三年后的春天，他回到了那个小镇..." |
| 视角转换 | 从另一个角色的视角看主角 | "酒馆老板还记得那个夜晚来的陌生人..." |
| 物件特写 | 聚焦一个物件，暗示后续 | "那把剑静静地挂在壁炉上方，积了薄薄的灰..." |
| 对话余音 | 一段未说出口的话或一封信 | "信封上写着她的名字，但始终没有寄出..." |
| 开放暗示 | 暗示故事可能还没有真正结束 | "远方的山脊上，有什么东西在闪光..." |

### 5.2 尾声禁忌

- ❌ 解释所有事情的发生原因（留白）
- ❌ 告诉玩家"你做了正确的选择"（让玩家自己判断）
- ❌ 预告续作（破坏当前故事的完整性）
- ❌ 过长（尾声应简洁，200 字以内为佳）

## §6 多周目与结局探索

### 6.1 结局提示

游戏结束后，可以给玩家结局探索提示（可选）：

```
你达成了结局: 「代价之路」(3/7)

还有其他可能的故事结局等待发现。
提示: 如果你在第三章选择信任 Marcus，故事可能会走向不同的方向...

要重新开始吗？(输入 /restart)
```

### 6.2 跨周目继承

| 继承类型 | 说明 | 示例 |
|----------|------|------|
| 记忆碎片 | 上周目的某个细节在新周目出现 | NPC 似曾相识的对话 |
| 结局加成 | 上周目的好结局给新周目一个小优势 | 起始物品/额外信息 |
| 隐藏路线 | 需要多周目才能解锁 | 第二周目才能发现的隐藏选项 |
| 元认知 | 游戏知道玩家玩过，对话有变化 | "你...好像来过这里？" |

## §7 与其他文档的关系

- **`branching-narrative.md`**: 分支叙事定义"怎么分叉"，结局系统定义"怎么收束"。结局是分支的终点。
- **`narrative-coherence.md`**: 结局必须通过一致性检查——不能与之前的选择矛盾。
- **`game-state-machine.md`**: 结局判定依赖游戏状态文件的完整记录。
- **`session-management.md`**: 结局后的存档管理——保留完整游戏记录供回顾。
- **`game-evaluation.md`**: 结局满意度是评估的核心指标之一。
- **`player-agency.md`**: 好的结局系统是"玩家自主权"的最终体现。
