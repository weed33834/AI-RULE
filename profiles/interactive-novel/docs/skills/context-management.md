# 上下文管理（Context Management）

> 本文档定义互动小说游戏引擎的上下文管理策略：多轮对话连贯性、关键决策后状态更新、跨会话上下文重建、token 预算分配与上下文压缩，确保长篇游戏中 AI 始终保持完整且准确的上下文。

## 1. 核心理念

上下文窗口是 AI 的"工作记忆"——它决定了 AI 当前能"看到"多少信息。互动小说游戏的长篇特性与有限的上下文窗口形成根本矛盾。本文档定义的策略，让 AI 在有限窗口内保持连贯叙事，不因上下文丢失而产生矛盾或断层。

上下文管理与记忆系统（`memory-system.md`）互补：
- **记忆系统**：负责"存"——将信息持久化到文件。
- **上下文管理**：负责"取"——决定什么信息进入当前上下文窗口。

## 2. 多轮对话连贯性策略

### 2.1 核心设定重注入

长篇游戏中，早期注入的系统设定会被后续对话挤出上下文。为保证 AI 始终"记得"游戏的基本框架，每 5 轮重注入一次核心设定。

**重注入频率**：每 5 轮一次（第 5、10、15、20...轮）。

**重注入内容**（精简版，约 300-500 token）：

```
[核心设定重注入 - 第 N 轮]
- 游戏类型：奇幻
- 世界观：中世纪魔法世界
- 主角：Roland，退役骑士
- 基调：史诗
- 难度：普通
- 交互模式：混合
- 当前目标：寻找王室密信
- 当前位置：橡木镇 老橡酒馆
- 玩家状态：HP 85/100，MP 30/50
- 关键 NPC：Marcus（好感25/信任10），Elara（好感-15/敌意10）
```

### 2.2 重注入触发条件

除了固定的每 5 轮一次，以下情况也触发重注入：

| 触发条件 | 重注入内容 | 原因 |
|----------|------------|------|
| 场景切换 | 新区域状态 + 区域内 NPC | 确保新场景描写准确 |
| 战斗开始 | 玩家战斗属性 + 敌人属性 | 确保战斗判定准确 |
| 重大剧情转折 | 更新后的目标 + 新的关键 NPC | 确保转折后方向正确 |
| 跨会话续玩 | 完整核心设定 + 剧情摘要 | 重建完整上下文 |

### 2.3 上下文锚点

除了定期重注入，以下信息作为"上下文锚点"始终保持在窗口中（不被压缩挤出）：

- **玩家当前 HP/MP**：每轮更新，始终可见。
- **当前位置**：场景切换时更新。
- **当前目标**：任务变化时更新。
- **最近 1 轮的叙事**：确保接续上轮内容。

```
上下文锚点（始终保留）：
┌─────────────────────────────────┐
│ [锚点] HP: 85/100 | MP: 30/50  │
│ [锚点] 位置: 老橡酒馆           │
│ [锚点] 目标: 寻找王室密信       │
│ [锚点] <最近1轮叙事原文>        │
└─────────────────────────────────┘
```

### 2.4 上下文优先级

当上下文窗口接近满载时，按优先级保留/丢弃信息：

| 优先级 | 内容 | 保留策略 |
|--------|------|----------|
| P0 | 系统指令 + 安全规则 | 永不丢弃 |
| P1 | 上下文锚点 | 永不丢弃 |
| P2 | 最近 3 轮对话 | 尽量保留 |
| P3 | 核心设定重注入 | 保留最新一次 |
| P4 | NPC 档案（当前交互中） | 保留当前 NPC |
| P5 | 早期对话 | 优先压缩（见 §5） |
| P6 | 工具输出 | 已沙盒化，不占窗口 |

## 3. 关键决策后状态更新

### 3.1 关键决策定义

以下事件属于"关键决策"，发生后必须立即更新状态文件：

| 事件类型 | 需更新的文件 | 更新内容 |
|----------|--------------|----------|
| 角色受伤/恢复 | `player.json` | HP/MP 当前值 |
| 获得/失去物品 | `player.json` → inventory | 物品列表 |
| NPC 关系变化 | `relationships.json` | 好感度/信任度等 |
| 任务状态变化 | `quests.json` | 新增/完成/失败 |
| 区域状态变化 | `world-map.json` | 区域状态字段 |
| 重大选择 | `choices-log.md` | 选择记录 |
| 重要事件 | `event-log.md` | 事件记录 |
| 时间推进 | `timeline.json` | 当前时间 |
| NPC 情绪变化 | `npcs/{name}.json` | 情感数值 |

### 3.2 状态更新流程

```
玩家做出关键决策（如：攻击 NPC）
        │
        ▼
判定结果（如：NPC 受伤，敌意+30）
        │
        ▼
立即更新所有相关状态文件：
├── player.json（如消耗了 MP/物品）
├── npcs/{name}.json（NPC 的 HP 和敌意值）
├── relationships.json（关系数值）
├── event-log.md（记录攻击事件）
├── choices-log.md（记录选择）
└── timeline.json（时间推进）
        │
        ▼
叙事生成时基于更新后的状态
        │
        ▼
本轮结束时验证状态与叙事一致性
```

### 3.3 状态更新原则

- **即时性**：关键决策后立即更新，不等本轮结束。
- **完整性**：所有受影响的状态文件都要更新，不遗漏。
- **可追溯**：每次更新记录原因（"因玩家攻击，敌意+30"）。
- **一致性**：叙事内容必须与状态文件一致，不得矛盾。

## 4. 跨会话上下文重建

### 4.1 重建流程

跨会话续玩时，上下文从零开始重建。详细流程见 `memory-system.md` §4，此处补充上下文层面的处理。

```
读取存档 + 设定 + 摘要 + 最近5轮
        │
        ▼
构建上下文窗口（按 token 预算分配）：
├── [20%] 系统指令（引擎规则 + 安全规则）
├── [15%] 核心设定（game-config.json 精简版）
├── [15%] 剧情摘要（story-summary.md 最新 2 段）
├── [20%] 最近 5 轮原文对话
├── [10%] 当前场景 NPC 档案
├── [10%] 当前玩家/世界状态
└── [10%] 生成空间
        │
        ▼
生成前情提要（2-3 段，展示给玩家）
        │
        ▼
玩家确认后，进入正常游戏循环
```

### 4.2 重建验证

上下文重建后，AI 自检：
- 核心设定是否完整？（类型/难度/角色/目标）
- 剧情摘要是否涵盖最新进展？
- 最近 5 轮是否准确还原？
- 当前状态是否与摘要一致？
- 相关 NPC 档案是否已加载？

### 4.3 信息缺失处理

如果存档不完整或文件缺失：
- 缺失非关键文件：用默认值填充，标注"信息缺失"。
- 缺失关键文件（player.json/game-config.json）：暂停，请玩家确认是否重新开始。
- 摘要与原文矛盾：以状态文件为准，修正摘要。

## 5. token 预算分配

### 5.1 标准预算分配（普通游戏，< 20 轮）

| 预算项 | 占比 | token 估算（8K 窗口） | 内容 |
|--------|------|----------------------|------|
| 系统指令 | 20% | 1600 | 引擎规则、安全规则、叙事原则 |
| 状态注入 | 15% | 1200 | 核心设定 + 当前状态锚点 |
| 最近对话 | 35% | 2800 | 最近 5-8 轮原文对话 |
| 档案加载 | 10% | 800 | 当前交互 NPC 档案 |
| 生成空间 | 15% | 1200 | 本轮叙事生成预留 |
| 缓冲区 | 5% | 400 | 应对突发需求 |

### 5.2 压缩预算分配（长篇游戏，20-50 轮）

| 预算项 | 占比 | 变化说明 |
|--------|------|----------|
| 系统指令 | 15% | 精简规则，保留核心 |
| 状态注入 | 15% | 不变 |
| 摘要层 | 15% | 新增：压缩后的剧情摘要 |
| 最近对话 | 25% | 减少到最近 3-5 轮 |
| 档案加载 | 10% | 不变 |
| 生成空间 | 15% | 不变 |
| 缓冲区 | 5% | 不变 |

### 5.3 极限预算分配（超长游戏，50 轮+）

| 预算项 | 占比 | 变化说明 |
|--------|------|----------|
| 系统指令 | 15% | 进一步精简 |
| L1 概览层 | 10% | 游戏设定 + 最新摘要 |
| L2 阶段层 | 15% | 近期阶段摘要 |
| L3 细节层 | 25% | 最近 5 轮原文 |
| L4 档案层 | 15% | 按需加载 |
| 生成空间 | 15% | 不变 |
| 缓冲区 | 5% | 不变 |

### 5.4 预算动态调整

token 预算不是固定的，根据实际情况动态调整：

| 场景 | 调整方向 | 原因 |
|------|----------|------|
| 战斗场景 | 档案层 +10%，对话层 -10% | 需要更多战斗属性数据 |
| NPC 深度对话 | 档案层 +10%，对话层 -10% | 需要 NPC 完整档案 |
| 探索场景 | 生成空间 +5%，对话层 -5% | 需要更多场景描写空间 |
| 上下文接近满载 | 触发紧急压缩 | 防止溢出 |

## 6. 上下文压缩策略

### 6.1 压缩触发条件

| 条件 | 压缩级别 |
|------|----------|
| 上下文使用率 > 70% | 轻度压缩 |
| 上下文使用率 > 80% | 中度压缩 |
| 上下文使用率 > 90% | 紧急压缩 |
| 对话轮数 ≥ 20 轮 | 常规压缩（见 `memory-system.md`） |

### 6.2 压缩级别

**轻度压缩（>70%）**：
- 压缩最早 3 轮对话为摘要。
- 保留所有关键决策和状态变化。
- 丢弃普通探索描写。
- 预计释放 15-20% 窗口空间。

**中度压缩（>80%）**：
- 压缩最早 5 轮对话为摘要。
- 合并连续的战斗描写。
- 保留最近 5 轮原文 + 所有摘要。
- 预计释放 25-30% 窗口空间。

**紧急压缩（>90%）**：
- 压缩所有非最近 3 轮的对话。
- 只保留：系统指令 + 锚点 + 摘要 + 最近 3 轮。
- 立即执行，不等轮次触发。
- 预计释放 40%+ 窗口空间。

### 6.3 压缩优先级（丢什么，留什么）

压缩时按以下优先级丢弃（从先到后）：

```
丢弃优先级（从高到低）：
1. 普通探索描写（"你走过走廊"）
2. 重复的战斗过程细节
3. 已解决的次要对话
4. 失败的无后果尝试
5. 早期 NPC 的寒暄对话
─── 以上优先丢弃 ───
6. 早期场景的环境描写（保留关键细节）
7. 非关键 NPC 的完整对话（保留摘要）
─── 谨慎丢弃 ───
8. 关键剧情决策（永远保留，写入摘要）
9. 玩家选择及后果（永远保留，写入 choices-log）
10. NPC 关系变化（永远保留，写入 relationships.json）
```

### 6.4 压缩后验证

压缩后必须验证：
- **信息完整性**：8 类必保信息是否都已在摘要中？
- **上下文锚点**：锚点信息是否仍在窗口中？
- **叙事连贯**：压缩后能否接续之前的叙事？
- **状态一致**：状态文件是否与压缩后上下文一致？

### 6.5 压缩熔断

- 连续 2 次压缩后上下文使用率仍 > 85%，触发压缩熔断。
- 输出"上下文压力报告"，建议玩家保存并开新会话。
- 不继续在高压上下文中生成，避免质量下降。

## 7. 上下文管理自检

每 5 轮执行上下文管理自检：

- [ ] 核心设定是否按规则重注入？（每 5 轮）
- [ ] 上下文锚点是否始终保留？
- [ ] 关键决策后状态是否即时更新？
- [ ] token 预算分配是否符合当前游戏阶段？
- [ ] 上下文使用率是否在安全范围（< 80%）？
- [ ] 压缩是否按规则执行？压缩后信息是否完整？

自检结果记录到 `.game-state/audit-log.md`。

### 7.5 AI Dungeon 式分层上下文组装（行业标杆）

> AI Dungeon 是 LLM 驱动无限叙事的架构范本。其分层上下文组装机制是处理"无限上下文窗口遗忘问题"的行业级方案。

### 分层结构

```
┌──────────────────────────────────────────┐
│ Required 层（必含，占总上下文 ≤70%）       │
│  ├─ 系统指令（引擎规则 + 安全规则）         │  P0 永不丢弃
│  ├─ 剧情要点（Plot Essentials）            │  P0 永不丢弃
│  ├─ 故事摘要（Story Summary）              │  P1 尽量保留
│  ├─ 作者注记（Author's Note）              │  P1 尽量保留
│  ├─ 前沿记忆（Front Memory）               │  P0 永不丢弃
│  └─ 最近动作（Last Action）                │  P0 永不丢弃
├──────────────────────────────────────────┤
│ Dynamic 层（动态填充剩余 token）           │
│  ├─ 世界设定卡（Story Cards / Lorebook）   │  ~25% 动态 token
│  │   按触发词的最近性 + 频率排序纳入        │
│  ├─ 记忆库（Memory Bank）                  │  ~25% 动态 token
│  │   每条 = 过去6个动作的AI摘要，按相关性检索│
│  └─ 故事历史（Story History）              │  ~50% 动态 token
│      从最近动作向前回溯直到填满             │
└──────────────────────────────────────────┘
```

### 最终上下文组装顺序

```
{系统指令}
{剧情要点}
世界设定: {Story Cards 按相关性排序...}
故事摘要: {最新摘要}
记忆: {Memory 1, 2, 3...}
近期故事: {Action 1 > User Input 1, Action 2 > User Input 2...}
[作者注记: {当前场景的风格/语气指引}]
> {最近用户输入} / {最近AI输出}
{前沿记忆: 当前最关键的状态提醒}
```

### 关键设计要点

| 要点 | 说明 |
|------|------|
| 故事卡触发 | 按关键词匹配 + 最近性 + 频率排序，最少看4条最近动作 |
| 记忆库检索 | 每条记忆是对过去6个动作的AI生成摘要，经嵌入存储按相关性检索 |
| 历史回溯 | 从最近动作向前回溯直到填满动态token预算 |
| 前沿记忆 | 当前最关键的1-3条状态提醒（如"角色手持武器""NPC正在愤怒"），始终完整包含 |
| 作者注记 | 当前场景的风格/语气指引，注入在历史之后、用户输入之前 |

### 与现有预算分配的关系

本节是对 §5 预算分配的深化。§5 定义了"各部分占多少"，本节定义了"各部分如何排序组装"。两者配合使用：
1. 先按 §5 计算各层 token 预算
2. 再按本节的组装顺序填充各层内容
3. Dynamic 层按相关性/最近性排序截断

## 8. 与其他系统的协作

| 协作系统 | 协作方式 |
|----------|----------|
| `memory-system.md` | 记忆系统负责存储，上下文管理负责读取和窗口分配 |
| `narrative-coherence.md` | 一致性检查依赖上下文中的完整信息 |
| `session-management.md` | 会话管理触发跨会话上下文重建 |
| `anti-dumb-ai.md` | 反弱智标准依赖上下文记忆的完整性 |
