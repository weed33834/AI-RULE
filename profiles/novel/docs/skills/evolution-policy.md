# 演进式精简策略 (Evolution Policy)

> 随着模型能力提升，很多规则会被模型默认遵守。过强的措辞反而限制 AI 在合理场景下的创作灵活性。
> 规则应该随模型升级而动态精简——但安全关键指令应该更紧而非更松。

## §1 规则分级与演进策略

| 规则等级 | 当前措辞 | 演进方向 | 触发条件 |
|----------|----------|----------|----------|
| 安全红线（P0） | "绝对禁止" | **不放松，反而更紧** | 每次发现新的安全 incident 后加固 |
| 核心行为（P2） | "必须""禁止" | **逐步放松为 "should"/"优先"** | 模型连续 10 次正确遵守后降级 |
| 效能指引（P2） | "鼓励""建议" | **可删除** | 模型通过 RLHF 默认具备此能力时 |
| 风格偏好（P2） | "用 X 而非 Y" | **可删除** | 模型输出已稳定符合偏好时 |

## §2 何时放松规则

### 可放松的信号
- 模型连续 10 次正确遵守该规则（无违反记录）。
- 该规则约束的行为已成为模型的 RLHF 默认行为。
- 模型升级后，新版模型在该规则上表现稳定。

### 放松方式
```
步骤 1: "绝对禁止 X" → "不应 X，除非有明确理由"
步骤 2: "不应 X" → "优先避免 X"
步骤 3: "优先避免 X" → 删除该规则
```

## §3 何时不放松规则

### 永不放松的规则
1. **内容分级 P0 红线**：未成年人保护、自残/自杀细节禁止——无论模型多强都不放松。
2. **密钥安全**：禁止硬编码密钥。
3. **MCP 红线**：禁止 AI 自下载/自配置 MCP。
4. **内部一致性**：角色 OOC、世界观矛盾、时间线错误——这是小说仓库的"真实性"底线。
5. **破坏性操作确认**：rm/git push 前确认。
6. **创作熔断**：连续失败后必须停止。

### 加固方式
```
每次发生创作 incident 后：
1. 在审计日志中记录 incident
2. 分析是否因规则不够严格导致
3. 如果是 → 增加更具体的约束
4. 在 CHANGELOG.md 中记录加固原因
```

## §4 创作技能生命周期

> 来源：Hermes Agent 技能自创建 + 创作迭代实践。

### 五阶段
| 阶段 | 触发条件 | 动作 |
|------|----------|------|
| 创建 | 完成复杂创作任务后 | 自动提取为可复用技能 |
| 使用 | 遇到类似创作任务时 | 自动加载匹配技能 |
| 评估 | 技能使用后 | 收集反馈、多维评估 |
| 改进 | 评估发现缺陷 | 优化技能文件 |
| 淘汰 | 技能过时或被替代 | 归档至 `docs/skills/archive/` |

### 评估指标
- 读者满意度：1-5 分反馈。
- 修改轮数：从初稿到定稿的迭代次数（越少越好）。
- 一致性违规次数：生成内容中的角色/情节/世界观矛盾数量。
- 陈词滥调检出率：被标记为模板化/套路化的表达比例。

> 以上指标的量化方法论见 `creative-evaluation.md` 模式 1（创意写作自动化评估）——三道判定 + 六维雷达图 + CI 集成，将"写得好不好"从主观判断升级为可量化、可回归的质量门禁。

## §5 创作技能策展器

> 来源：Hermes Curator + 创作技法更新机制。

### 每月扫描
- 检测过时技能：叙事技法演变、类型惯例更新、读者口味变化。
- 合并重复技能：多个技能处理同一创作任务的不同变体时建议合并。
- 安全约束：策展器**只建议不执行**，所有变更必须经作者确认。

### 策展报告格式
```markdown
# 创作技能策展报告
## 日期: YYYY-MM-DD
### 需要更新的技能
- [技能名]: [更新原因] → [新技法来源]
### 可合并的技能
- [技能A] + [技能B]: [合并理由]
### 建议归档的技能
- [技能名]: [淘汰原因] → [替代方案]
```

## §6 创作轨迹洞察

> 来源：Amazon Bedrock AgentCore + 创作质量研究。

### 轨迹数据与规则精简的闭环
```
创作轨迹数据 → 轨迹洞察分析 → 规则/技能调整建议 → 作者确认 → 更新 → 新轨迹数据
```

### 沉默失败检测与规则加固
- 检测到"角色 OOC"频发 → 加固角色一致性检查规则。
- 检测到"伏笔遗忘"频发 → 加固伏笔追踪规则。
- 检测到"陈词滥调"频发 → 加固去 AI 文学味规则。

## 核心原则
- 安全红线永不放松；创作灵活性规则可随模型进步逐步精简。
- 每季度执行一次精简检查：读审计日志 → 统计违反次数 → 放松零违反规则 → 加固 incident 相关规则。
