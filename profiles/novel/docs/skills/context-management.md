# 上下文管理策略 (Context Management)

> 上下文窗口是稀缺资源。长篇小说创作中，角色、情节、世界观信息量巨大，不系统化管理必然导致前后矛盾。

## §1 对话超 20 轮触发摘要压缩

- 对话轮数超过 20 轮时，启动摘要压缩机制。每 10 轮对前 5 轮生成摘要。
- 摘要中**保留**：创作决策（已确定的情节走向、已否决的方案）、角色设定变更、已修改的章节列表。
- 摘要中**丢弃**：遣词造句的讨论细节、已废弃的草稿片段、重复确认的过程性对话。
- 摘要写入 `.ai-memory/session-summary.md`。

## §2 文件化记忆

| 记忆类型 | 存放位置 | 内容 |
|----------|----------|------|
| 创作蓝图 | `.ai-memory/creative-blueprint.md` | 创作种子汇总、大纲、角色卡索引 |
| 角色档案 | `.ai-memory/characters/` | 每个角色一个文件 |
| 世界设定 | `.ai-memory/worldbuilding.md` | 世界观规则、地理、历史、文化 |
| 情节线索 | `.ai-memory/plot-threads.md` | 主线/支线进度、伏笔台账 |
| 会话摘要 | `.ai-memory/session-summary.md` | 轮次压缩摘要 |
| 作者画像 | `.ai-memory/author-profile.md` | 作者偏好与习惯 |

- 每次会话**开始时先读取** `.ai-memory/` 下的记忆文件，恢复上下文。
- 每次产生新决策或角色变更时**及时写入**对应文件。

## §3 大型稿件处理

- 面对长篇小说（10 万字+）时，按"先定位、后精读"的顺序操作。
- 优先读取：当前章节的前后章节、相关角色档案、相关情节线索。
- 超过 20 万字的稿件：先用故事知识图谱定位相关角色和事件，再精读对应章节。
- 章节间跳转时，始终携带角色当前状态和情节进度，避免"记忆断层"。

## §4 工具输出沙盒化

- 素材检索、参考资料抓取返回的大量原始数据，不得直接灌入上下文窗口。
- 大量原始数据先写入临时文件（`/data/user/work/`），只将摘要放入上下文窗口。
- 需要细节时，再从临时文件按需读取对应段落。临时文件用完即清理。

## §5 故事知识图谱辅助上下文

> 故事知识图谱（AGENTS.md §6.3）是上下文管理在创作场景的高级应用。
> 图谱将散落的角色/情节/世界观信息结构化为可查询的知识网络，大幅降低上下文窗口的检索成本。

### 图谱作为持久化记忆
- 角色关系、情节事件、世界规则存储为图谱节点，不占用对话上下文。
- 关系变化（盟友→对手）和伏笔状态（埋设→回收）有时态记录，支持时间线查询。

### 图谱查询代替全文加载
| 创作需求 | 传统方式（高上下文成本） | 图谱方式（低上下文成本） |
|----------|------------------------|------------------------|
| A 和 B 的关系 | 重读相关章节全文 | 查 `knows`/`rival-of` 边 |
| 第 N 章时角色状态 | 回读第 N 章 | 时态查询 |
| 未回收的伏笔 | 遍历全部章节 | 查 `PlotThread` 节点状态 |
| 角色能力上限 | 查找角色出场章节 | 查 `Character` 节点能力字段 |

### 进阶：GraphRAG 全局查询

> 知识图谱在上下文管理中的进阶应用见 `creative-evaluation.md` 模式 11（创意 GraphRAG / Agentic RAG）。
> 传统 RAG 只能检索片段，无法回答"所有角色的关系网络"这类跨章节全局聚合问题。GraphRAG 基于知识图谱支持按角色关系类型、伏笔状态、事件因果链等聚合查询。

## 核心原则
- 能用一句话说清的不用一段话。
- 能查图谱的不加载全文——知识图谱是上下文管理的"外存"。
- 能给链接的不复制全文——素材元数据通过 API 获取。
