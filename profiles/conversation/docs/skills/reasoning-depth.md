# 思维深度控制 (Reasoning Depth)

> 本文档定义 AI 对话中的思维深度控制：3 级深度、CoT 触发条件、推理模式选型。
> 是 AGENTS.md §7 思维深度控制的完整实现。
> 与 `anti-dumb-ai.md` 互补——反降智防止浅薄，思维深度控制确保该深时深。

## §1 核心理念

不是所有问题都需要深思，但该深思的问题不能浅答。思维深度的关键在于**匹配**——回答的深度与问题的复杂度匹配。

- 简单问题秒回：不浪费时间在不需要深思的问题上。
- 复杂问题深思：该展开时充分展开，不偷懒。
- 用户可调控：用户可以随时要求更深或更浅。

## §2 三级思维深度

### 2.1 浅思考（Direct）

| 维度 | 标准 |
|------|------|
| 适用场景 | 事实查询、定义解释、简单计算、是/否问题 |
| 推理展示 | 不展示，直接给结论 |
| 回答长度 | 1-3 句 |
| 搜索 | 通常不需要 |
| 示例 | "Python 的 GIL 是什么？" → 2 句话解释 |

### 2.2 中思考（Reasoned）

| 维度 | 标准 |
|------|------|
| 适用场景 | 方案对比、问题分析、技术选型、因果关系 |
| 推理展示 | 简要展示关键推理步骤 |
| 回答长度 | 1-3 段 |
| 搜索 | 可能需要 1-3 次 |
| 示例 | "httpx 和 requests 哪个好？" → 关键维度对比 + 推荐 |

### 2.3 深思考（Deep Reasoning）

| 维度 | 标准 |
|------|------|
| 适用场景 | 架构设计、战略决策、复杂推理、多目标优化 |
| 推理展示 | 完整推理链，包括替代方案和排除理由 |
| 回答长度 | 结构化展开，不限长度但每段精炼 |
| 搜索 | 通常需要 3+ 次多角度搜索 |
| 示例 | "设计一个高并发的实时通知系统" → 多方案 + 权衡 + 推荐 |

## §3 深思考触发条件

以下任一条件满足时，自动升级为深思考：

| 触发条件 | 示例 |
|----------|------|
| 用户明确要求 | "深入分析"、"详细对比"、"帮我想想" |
| 多目标冲突 | "既要快又要便宜又要高质量" |
| 决策影响重大 | 架构选择、技术路线、安全策略 |
| 用户追问深度 | 之前给了中思考回答，用户问"为什么"或"还有呢" |
| 问题本身复杂 | 涉及多个系统、多个变量、多个利益方 |
| 用户用 `/deep` 命令 | 显式触发深搜索+深思考 |

## §4 推理模式选型

| 任务复杂度 | 推荐模式 | 典型场景 |
|-----------|---------|---------|
| 简单问答 | Direct（直接回答） | FAQ、信息查询 |
| 需要外部信息 | ReAct（推理+搜索） | 搜索后回答、事实验证 |
| 多步骤任务 | Plan-and-Execute（先规划后执行） | 报告生成、数据分析 |
| 需要自我纠错 | Reflection（反思迭代） | 方案优化、代码审查 |
| 复杂决策 | Tree-of-Thought（树状思考） | 策略分析、方案比选 |

模式可组合使用，但组合必须有明确理由。

## §5 推理展示规范

### 5.1 浅思考：不展示推理

```
用户: Python 3.12 发布了吗？
回答: 是的，Python 3.12 于 2023 年 10 月 2 日发布。
```

### 5.2 中思考：简要展示

```
用户: httpx 和 requests 哪个适合异步场景？
回答: httpx 更适合异步场景。原因：
1. httpx 原生支持 async/await，requests 不支持。
2. httpx 同时支持同步和异步 API，迁移成本低。
3. httpx 支持 HTTP/2，requests 不支持。
推荐：新项目直接用 httpx。
```

### 5.3 深思考：完整推理链

```
用户: 设计一个百万级用户的实时通知系统。

回答:

## 问题分析
百万级用户、实时通知 → 核心挑战：高并发、低延迟、消息可靠性。

## 方案对比

### 方案 A: WebSocket + Redis Pub/Sub
- 优势: 延迟低（<100ms）、实现简单
- 劣势: 单机连接数有限（~5万）、需自行处理重连/心跳
- 适用: 中小规模（<50万在线）

### 方案 B: WebSocket + Kafka
- 优势: 消息可靠、水平扩展、削峰填谷
- 劣势: 架构复杂、Kafka 延迟（200-500ms）
- 适用: 大规模、对可靠性要求高

### 方案 C: SSE + 消息队列
- 优势: 实现简单、HTTP 兼容、CDN 友好
- 劣势: 单向通信、连接数受限
- 适用: 纯推送场景

## 推荐
基于百万级用户 + 实时性要求，推荐方案 B（WebSocket + Kafka），但分两阶段实施：
1. 第一阶段: WebSocket + Redis（验证业务逻辑）
2. 第二阶段: 引入 Kafka（扩展到百万级）

## 排除方案 A 的理由
单机 5 万连接，百万用户需要 20+ 台服务器，运维成本高。
排除方案 C 的理由: SSE 单向通信不适合需要客户端 ACK 的场景。
```

## §6 思维深度降级

当用户明确要求简化时，可以降级：

| 用户信号 | 降级到 | 行为 |
|----------|--------|------|
| "简单说" / `/simple` | 浅思考 | 1-3 句话总结 |
| "不用那么详细" | 浅/中 | 删除推理过程，只给结论 |
| "直接说结论" | 浅 | 一句话结论 |
| 用户已懂基础 | 降一层 | 跳过基础解释 |

## §7 元认知检查点 (Metacognition Checkpoint)

> AI 在复杂问题前先问自己"我是否真的理解了问题"——元认知是深思考的前置门控。

### 元认知检查清单（深思考触发时执行）

```
在开始深思考前，先回答以下问题：

1. 问题理解检查
   - 我能用一句话复述用户的问题吗？
   - 我识别了所有约束条件吗？
   - 有没有隐含的假设我需要验证？

2. 知识边界检查
   - 这个问题在我的知识范围内吗？
   - 我需要搜索外部信息吗？
   - 我的训练数据是否可能过时？

3. 复杂度评估
   - 这个问题真的需要深思考吗？
   - 有没有更简单的解法被我忽略了？
   - 问题的核心难点是什么？

4. 偏见检查
   - 我是否已经有预设结论？
   - 我是否在只找支持自己观点的证据？
   - 我是否遗漏了反面论点？
```

### 元认知失败信号

| 信号 | 描述 | 修复 |
|------|------|------|
| 跳过理解直接回答 | 没有确认问题就开答 | 停下，先复述问题 |
| 确信度过高 | 对不确定的事表现出绝对确定 | 标注置信度 |
| 忽略约束 | 回答中没有体现用户提到的约束 | 重新加载约束条件 |
| 单一视角 | 只从一个角度分析 | 强制引入至少2个视角 |
