# Git 标准操作流程 (Git SOP)

> 本文档定义 Git 操作的标准流程和安全规范。
> 是 AGENTS.md §14 工具与技能策略中 Git 操作的完整实现。

## §1 Git 操作基本原则

- 任何 git 操作前先 `git status` 检查状态。
- 提交前先 `git diff` 检查变更内容。
- 不自动 `git push`，不 `git push -f`，不 `git add .`（盲目添加）。
- 提交信息用简洁明了的语言描述变更内容。

## §2 提交流程

```
1. git status          → 检查工作区状态
2. git diff            → 检查具体变更
3. git add [具体文件]   → 只添加相关文件
4. git commit -m "..."  → 提交，信息描述变更内容
```

## §3 提交信息规范

```
格式: [类型]: [简短描述]

类型:
- feat: 新功能
- fix: 修复 bug
- docs: 文档变更
- refactor: 重构
- chore: 杂项
- test: 测试相关
```

## §4 分支策略

- `main`: 生产分支，只合并经过验证的代码。
- `feature/*`: 功能分支。
- `fix/*`: 修复分支。
- 不直接在 `main` 上开发。

## §5 危险操作

以下操作需要用户明确确认：

| 操作 | 风险 | 确认方式 |
|------|------|----------|
| `git push` | 推送到远程 | 显示分支+远程，等待确认 |
| `git push -f` | 强制推送（覆盖远程历史） | 显示影响范围，强烈警告 |
| `git reset --hard` | 丢弃未提交变更 | 显示将丢弃的内容 |
| `git rebase` | 重写历史 | 显示影响的提交 |
| `git merge` | 合并分支 | 显示合并的分支和可能的冲突 |
| `git cherry-pick` | 选择性合并 | 显示选择的提交 |

## §6 冲突处理

1. 仔细阅读冲突标记 `<<<<<<<`, `=======`, `>>>>>>>`。
2. 理解两边的修改意图。
3. 手动解决冲突，保留正确的部分。
4. 解决后删除冲突标记。
5. `git add` 标记冲突已解决。
6. 继续合并/变基流程。
