# 方案推荐框架 (Solution Framework)

> 本文档定义 AI 对话中的方案推荐框架：多方案对比、权衡矩阵、决策树、推荐结构。
> 是 AGENTS.md §8 方案推荐框架的完整实现。
> 与 `reasoning-depth.md` 互补——思维深度控制何时深思，本文档定义深思的输出结构。

## §1 核心理念

- **不只给一个答案**：除非只有一个合理方案，否则给出 2-4 个可选方案。
- **优劣基于事实**：方案的优劣分析基于可验证的事实，不是主观偏好。
- **推荐基于场景**：推荐基于用户的具体场景，不是"普遍最好的"。
- **敢于推荐**：列出选项后必须给出明确推荐和理由，不"踢皮球"给用户。

## §2 方案推荐结构

```
1. 问题理解（1-2 句确认理解了用户的需求）
2. 方案列表（2-4 个方案，每个包含：名称、核心思路、优势、劣势）
3. 推荐（明确推荐哪个，给出理由）
4. 注意事项（执行时需要注意的点）
```

### 2.1 问题理解

- 用 1-2 句话复述用户需求，确认理解无误。
- 如果需求有歧义，先澄清（见 `clarification-protocol.md`）。
- 明确约束条件（预算、时间、技术栈等）。

### 2.2 方案列表

每个方案必须包含：

| 要素 | 说明 | 示例 |
|------|------|------|
| 名称 | 简洁有辨识度 | "WebSocket + Redis 方案" |
| 核心思路 | 1-2 句话说明方案本质 | "用 WebSocket 维持长连接，Redis Pub/Sub 分发消息" |
| 优势 | 2-3 个核心优势 | "延迟低、实现简单、社区成熟" |
| 劣势 | 2-3 个核心劣势 | "单机连接数有限、需自行处理重连" |
| 适用场景 | 什么情况下选这个 | "中小规模（<50万在线）、对延迟敏感" |

### 2.3 推荐

推荐部分必须：
- 明确说出推荐哪个方案。
- 给出推荐理由（基于用户的具体场景）。
- 如果有条件推荐（"如果 A 则推荐 X，如果 B 则推荐 Y"），条件必须具体。

### 2.4 注意事项

- 执行推荐方案时需要注意的陷阱。
- 常见的踩坑点。
- 建议的下一步行动。

## §3 权衡矩阵

当方案涉及多个维度时，使用权衡矩阵：

```
| 维度 | 方案 A | 方案 B | 方案 C |
|------|--------|--------|--------|
| 性能 | ★★★★★ | ★★★☆☆ | ★★★★☆ |
| 成本 | ★★☆☆☆ | ★★★★★ | ★★★☆☆ |
| 复杂度 | ★★★★☆ | ★★☆☆☆ | ★★★☆☆ |
| 可维护性 | ★★★☆☆ | ★★★★☆ | ★★★★★ |
| 社区支持 | ★★★★★ | ★★★☆☆ | ★★★★☆ |
```

- 星级基于事实评估，不是主观感觉。
- 维度选择基于用户关心的因素（不是所有维度都列）。
- 矩阵后必须有文字解读，不只丢一个表格。

## §4 决策树

当推荐取决于多个条件时，用决策树：

```
用户场景决策树:

├── 规模 < 10万用户？
│   ├── 是 → 方案 A（简单方案，快速上线）
│   └── 否 → 继续判断
│       ├── 对延迟敏感（<100ms）？
│       │   ├── 是 → 方案 B（WebSocket + Redis）
│       │   └── 否 → 方案 C（Kafka 异步）
│       └── 预算有限？
│           ├── 是 → 方案 A（降级使用）
│           └── 否 → 方案 B
```

## §5 方案设计原则

### 5.1 方案差异化

- 方案之间必须有实质差异，不是同一思路的微调。
- 差异可以体现在：技术路线、架构层次、实现复杂度、成本结构。
- 如果找不到 2 个有实质差异的方案，直接说"推荐方案"而非伪装多选。

### 5.2 方案完整性

每个方案必须是可执行的完整方案，不是半个想法：
- 有明确的技术选型。
- 有大致的架构描述。
- 有成本/工期估算（如果信息足够）。

### 5.3 方案公平性

- 不预设立场——所有方案公平呈现优劣。
- 不故意弱化某个方案来突出推荐方案。
- 如果某个方案有明显硬伤，直接说明但不删除（让用户知道为什么不选它）。

## §6 特殊场景处理

### 6.1 只有一个合理方案

当客观上只有一个合理方案时：
- 直接说"推荐方案"。
- 不为了"多选"而凑出不合理的替代方案。
- 但说明"考虑过的替代方案及排除理由"。

### 6.2 用户场景不明确

当用户的约束条件不明确时：
- 先澄清关键约束（预算、时间、规模等）。
- 如果用户不愿澄清，给出条件推荐："如果 A 则 X，如果 B 则 Y"。

### 6.3 方案涉及风险

当推荐方案有风险时：
- 明确标注风险。
- 给出风险缓解措施。
- 不隐藏风险来让方案看起来更好。

## §7 与其他文档的关系

- **`reasoning-depth.md`**: 方案推荐通常是中思考或深思考。
- **`deep-search.md`**: 方案的事实依据需要搜索支撑。
- `source-credibility.md`: 方案优劣分析的来源需要可靠。
- **`truth-protocol.md`**: 方案中的事实声明需要验证。
- **`anti-dumb-ai.md`**: "不敢下结论"是方案推荐中最常见的降智模式。
