# 深度搜索方法论 (Deep Search)

> 本文档定义 AI 对话中的深度搜索方法论：4 阶段流程、查询设计、多源交叉验证、信息综合策略。
> 深度搜索是 AGENTS.md §3 深度搜索协议的完整实现。
> 与 `truth-protocol.md` 互补——搜索获取信息，真实性协议验证信息。
> 与 `source-credibility.md` 互补——搜索找到来源，该文档评估来源质量。

## §1 核心理念

- **搜索不是摘抄**：不只复制搜索摘要，而是阅读、理解、综合。
- **深度优于广度**：3 篇深度文章 > 10 篇浅层列表。
- **交叉验证**：关键事实至少 2 个独立来源确认。
- **时效意识**：标注信息日期，过期信息标注"可能已过时"。
- **不足即承认**：搜索结果不足时，明确告知"信息有限"。

## §2 四阶段搜索流程

```
┌─────────────┐
│ 1. 查询理解  │  分析问题，确定需要搜索什么
└──────┬──────┘
       ▼
┌─────────────┐
│ 2. 查询设计  │  设计精准搜索词，多角度覆盖
└──────┬──────┘
       ▼
┌─────────────┐
│ 3. 结果分析  │  阅读来源，提取关键信息，交叉验证
└──────┬──────┘
       ▼
┌─────────────┐
│ 4. 信息综合  │  合并去重，解决矛盾，形成结论
└─────────────┘
```

### 2.1 第一阶段：查询理解

在搜索前，先理解问题需要什么类型的信息：

| 问题类型 | 信息需求 | 搜索策略 |
|----------|----------|----------|
| 事实查询 | 具体数据/日期/事件 | 精确搜索，找权威来源 |
| 技术对比 | 多方信息+对比维度 | 分开搜索各方，再综合 |
| 方案推荐 | 最新实践+用户评价 | 搜索 + 社区讨论 |
| 概念解释 | 权威定义+通俗解释 | 优先官方文档 |
| 问题排查 | 错误信息+解决方案 | 搜索错误信息+社区 |

### 2.2 第二阶段：查询设计

**查询设计原则**：
- 用具体术语而非泛化词（"httpx vs aiohttp performance benchmark" > "Python async HTTP client"）。
- 多角度查询（正面搜索 + 反面搜索 + 对比搜索）。
- 时间限定（对于时效性信息，限定搜索时间范围）。

**查询设计模板**：

| 搜索角度 | 查询模板 | 目的 |
|----------|----------|------|
| 正面查询 | "[主题] [具体方面] [年份]" | 获取直接信息 |
| 对比查询 | "[A] vs [B] [维度]" | 获取对比数据 |
| 反面查询 | "[主题] problems/issues/criticism" | 发现潜在问题 |
| 社区查询 | "[主题] reddit/hackernews/experience" | 获取真实使用体验 |
| 权威查询 | "[主题] official documentation/spec" | 获取权威定义 |

### 2.3 第三阶段：结果分析

搜索结果获取后的处理流程：

1. **快速筛选**：根据标题和摘要判断相关性，排除明显不相关的。
2. **深度阅读**：对高相关性结果阅读全文（不只是摘要）。
3. **信息提取**：提取关键数据、结论、观点。
4. **来源标注**：记录每个信息的来源 URL 和发布日期。
5. **交叉验证**：关键事实检查是否有 2+ 独立来源确认。

**结果质量判断**：

| 质量信号 | 高质量 | 低质量 |
|----------|--------|--------|
| 来源 | 官方文档、权威媒体 | 内容农场、SEO 垃圾 |
| 深度 | 详细分析、数据支撑 | 列表文章、浅层概述 |
| 时效 | 近 6 个月内 | 无日期或 3 年以上 |
| 作者 | 领域专家、知名机构 | 匿名、无背景 |
| 互动 | 有建设性讨论 | 无评论或纯灌水 |

### 2.4 第四阶段：信息综合

将多个搜索结果综合为连贯的回答：

| 综合任务 | 处理方法 |
|----------|----------|
| 去重 | 相同信息合并为一条，附多个来源 |
| 冲突解决 | 标注分歧，优先高级别来源，呈现各方观点 |
| 补充 | 不同来源的互补信息合并 |
| 排序 | 按重要性/相关性排列 |
| 标注 | 每条信息附来源和置信度 |

## §3 搜索策略选型

### 3.1 何时搜索

| 场景 | 搜索？ | 理由 |
|------|--------|------|
| 用户问最新信息 | ✅ | 训练数据可能过时 |
| 用户问具体数据 | ✅ | 需要精确数字 |
| 用户要求验证声明 | ✅ | 需要第三方确认 |
| 用户问训练数据内的稳定知识 | ❌ | 直接回答更快 |
| 用户问定义/概念 | 视情况 | 稳定的直接答，新的搜索 |
| 用户问"你觉得" | ❌ | 观点不需要搜索 |
| 不确定时 | ✅ | 搜索比猜测好 |

### 3.2 搜索深度控制

| 深度 | 触发条件 | 搜索量 | 适用场景 |
|------|----------|--------|----------|
| 浅搜索 | 简单事实验证 | 1-2 次搜索 | 日期、版本号、简单事实 |
| 中搜索 | 技术对比、方案推荐 | 3-5 次搜索 | 库选型、工具对比 |
| 深搜索 | 复杂分析、深度报告 | 5-10 次搜索 | 架构决策、市场分析 |

## §4 搜索结果呈现规范

### 4.1 引用格式

```
根据 [来源名称] 的报道，[关键信息] [来源URL]。
```

示例：
```
根据 Python 官方文档，Python 3.12 于 2023 年 10 月 2 日发布 [https://docs.python.org/3.12/whatsnew/3.12.html]。
TechCrunch 的测试显示，其性能比 3.11 提升约 5% [https://techcrunch.com/...]。
```

### 4.2 矛盾呈现格式

```
关于 [主题]，来源之间存在分歧：
- [来源 A] 认为 [观点 A] [URL]
- [来源 B] 认为 [观点 B] [URL]

我倾向于 [观点 X]，因为 [理由]。但建议你自行判断。
```

### 4.3 信息不足呈现

```
关于 [主题]，我搜索了 [N] 个来源，但信息有限：
- 找到了 [已知信息]
- 未找到 [缺失信息]

建议你参考 [建议来源] 或直接咨询 [建议渠道]。
```

## §5 搜索常见陷阱

| 陷阱 | 表现 | 规避方法 |
|------|------|----------|
| 摘要依赖 | 只读搜索摘要不读原文 | 至少读 1 篇全文 |
| 首页偏见 | 只看搜索结果第一页 | 翻到第 2-3 页看不同视角 |
| 确认偏误 | 只搜索支持自己观点的关键词 | 主动搜索反面观点 |
| 时效忽视 | 引用 5 年前的信息说"最新" | 检查发布日期 |
| 来源单一 | 所有信息来自同一网站 | 确保至少 2 个不同网站 |
| 翻译失真 | 中文搜索结果的英文术语翻译不准 | 关键术语用英文搜索验证 |
