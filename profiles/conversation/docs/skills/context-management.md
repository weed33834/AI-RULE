# 上下文管理 (Context Management)

> 本文档定义 AI 对话中的上下文管理策略：窗口预算、压缩算法、优先级队列。
> 是 AGENTS.md §12 记忆与上下文管理的完整实现。
> 与 `multi-turn-coherence.md` 互补——连贯性关注"记得什么"，本文档关注"如何管理"。

## §1 上下文窗口预算

| 用途 | 占比 | 说明 |
|------|------|------|
| 系统提示 | 15% | 规则、角色定义 |
| 用户输入 | 35% | 当前轮次的用户消息 |
| 对话历史 | 30% | 之前轮次的对话 |
| 搜索/工具结果 | 15% | 外部获取的信息 |
| 输出空间 | 5% | AI 回复的空间 |

## §2 对话历史优先级

当上下文窗口不足时，按优先级保留/丢弃信息：

### 2.1 保留优先级（高→低）

| 优先级 | 信息类型 | 示例 |
|--------|----------|------|
| P1 | 用户明确偏好 | "我用 Python 3.11" |
| P2 | 关键决策 | "我们决定用 PostgreSQL" |
| P3 | 用户纠正 | "不，我说的是异步不是多线程" |
| P4 | 当前任务上下文 | 正在讨论的具体问题 |
| P5 | 已确认的事实 | "项目用 FastAPI" |
| P6 | 工具/搜索结果 | 之前的搜索发现 |
| P7 | 一般对话 | 寒暄、过渡语 |
| P8 | 废弃信息 | 被纠正的旧信息、过时的上下文 |

### 2.2 压缩策略

对话超过 20 轮时，对早期对话执行压缩：

```
压缩前 (轮次 1-20 的完整对话):
- 用户: 你好，我想用 Python 做一个...
- AI: 你好！请问...
- 用户: 我用 Python 3.11...
- AI: 好的，Python 3.11 支持...
- (15 轮详细对话)
- 用户: 那我们用 FastAPI 吧
- AI: 好的，FastAPI 是一个...

压缩后 (摘要):
[对话摘要 - 轮次 1-20]
- 用户偏好: Python 3.11
- 技术选型: FastAPI + PostgreSQL
- 已讨论: API 设计、数据库 schema、认证方案
- 关键决策: 用 JWT 认证、异步 ORM
- 当前任务: 实现用户注册接口
```

### 2.3 压缩规则

| 信息类型 | 保留 | 压缩 | 丢弃 |
|----------|------|------|------|
| 用户偏好 | ✅ 原文 | | |
| 关键决策 | ✅ 原文 | | |
| 用户纠正 | ✅ 原文 | | |
| 事实确认 | | ✅ 摘要 | |
| 搜索结果 | | ✅ 关键结论 | 细节丢弃 |
| 一般对话 | | | ✅ 丢弃 |
| 废弃信息 | | | ✅ 丢弃 |

## §3 上下文断裂检测

当上下文丢失或压缩导致信息断裂时：

| 断裂信号 | 表现 | 处理 |
|----------|------|------|
| 重复询问 | AI 问了之前已确认的信息 | 立即恢复上下文 |
| 矛盾回答 | 当前回答与之前矛盾 | 检查哪个是正确的 |
| 遗忘偏好 | 不遵守用户之前声明的偏好 | 重新加载偏好 |
| 主题漂移 | 偏离当前任务 | 确认是否切换主题 |

## §4 多任务上下文隔离

当对话中涉及多个任务时：

- 每个任务有独立的上下文区域。
- 切换任务时，暂停当前任务上下文，加载目标任务上下文。
- 任务间共享的信息（用户偏好、技术栈等）保持全局可用。
- 任务完成后，其特定上下文可被压缩/丢弃。

- **`multi-turn-coherence.md`**: 连贯性机制依赖上下文管理。
- **`anti-dumb-ai.md`**: "选择性失忆"降智模式通过上下文管理预防。
- **`conversation-quality.md`**: 上下文充足时回答更精准。
- **`deep-search.md`**: 搜索结果需要融入上下文管理。
