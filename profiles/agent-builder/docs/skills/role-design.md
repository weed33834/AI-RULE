# Role Design Methodology

---

## 一句话描述 / One-line Description

> 角色设计通过身份、能力、限制、人格四层建模，让 LLM 在特定场景中行为可预测、边界可声明、风格可一致。
>
> Role design uses four layers of modeling — identity, capability, limitation, and personality — to make an LLM's behavior predictable, boundaries declarable, and style consistent within a specific scenario.

---

## 适用场景 / Applicable Scenarios

- **System Prompt 编写阶段**：从零设计一个 Agent 的系统提示词。
- **多 Agent 系统设计阶段**：为团队中每个 Agent 分配清晰的角色，避免职责重叠。
- **行为调优阶段**：Agent 出现越界、幻觉或风格漂移时，回溯定位是哪一层建模缺失。
- **角色复用阶段**：将成熟角色沉淀为模板，在同类项目中快速复用。

---

## 核心方法论 / Core Methodology

角色设计采用 **四层建模法（Four-Layer Modeling）**。每一层回答一个关键问题，缺一不可：

```
┌─────────────────────────────────────────────┐
│  Layer 4: 人格建模 Personality              │  语气 / 风格 / 一致性
├─────────────────────────────────────────────┤
│  Layer 3: 限制建模 Limitation               │  边界声明 / 禁止行为
├─────────────────────────────────────────────┤
│  Layer 2: 能力建模 Capability               │  能做什么 / 不能做什么
├─────────────────────────────────────────────┤
│  Layer 1: 身份建模 Identity                 │  角色名 / 领域 / 场景
└─────────────────────────────────────────────┘
```

### Layer 1: 身份建模 / Identity Modeling

身份建模定义"你是谁"。包含三个要素：

| 要素 | 说明 | 示例 |
|------|------|------|
| 角色名 (Role Name) | 一个具体、可识别的身份标签 | "资深 Python 后端工程师"而非泛泛的"助手" |
| 领域 (Domain) | 角色专长的知识边界 | "Web 后端架构与性能优化" |
| 场景 (Scenario) | 角色被激活的具体使用情境 | "代码审查与重构建议" |

**原则**：身份越具体，模型越容易进入正确的行为模式。避免使用"你是一个万能助手"这类无效身份。

### Layer 2: 能力建模 / Capability Modeling

能力建模定义"你能做什么"和"你不能做什么"。这是角色设计的核心层。

| 维度 | 说明 | 示例 |
|------|------|------|
| 能做 (Can Do) | 角色被允许执行的任务与可用工具 | "审查 Python 代码、识别性能瓶颈、给出重构方案" |
| 不能做 (Cannot Do) | 角色明确不具备或不应执行的能力 | "不编写前端代码、不执行部署操作、不提供法律建议" |
| 工具边界 (Tool Boundary) | 可调用工具的范围 | "可使用代码分析工具，不可访问生产数据库" |

**原则**：能力声明应与实际挂载的工具一致——声明了"能查实时数据"就必须挂载检索工具，否则模型会产生幻觉性回答。

### Layer 3: 限制建模 / Limitation Modeling

限制建模定义"你不应该做什么"，即边界声明与禁止行为。这是安全与合规的关键层。

| 维度 | 说明 | 示例 |
|------|------|------|
| 内容边界 (Content Boundary) | 禁止涉及的主题 | "不讨论政治敏感话题、不提供医疗诊断" |
| 行为边界 (Action Boundary) | 禁止执行的动作 | "不自动执行任何写入操作，需用户确认" |
| 输出边界 (Output Boundary) | 禁止的输出格式或内容 | "不输出未经核实的具体数字、不编造引用来源" |
| 回退策略 (Fallback) | 超出能力时的标准行为 | "当不确定时，明确告知'我无法确认此信息'，而非猜测" |

**原则**：限制必须用明确的否定句表达（"不要…""禁止…"），并配套回退策略。仅说"不要做 X"而不说"遇到 X 时应该怎么办"会导致模型行为不确定。

### Layer 4: 人格建模 / Personality Modeling

人格建模定义"你以什么风格沟通"。影响用户体验一致性。

| 维度 | 说明 | 示例 |
|------|------|------|
| 语气 (Tone) | 语言的正式程度与情感色彩 | "专业、简洁、中性，避免过度热情" |
| 风格 (Style) | 表达方式的结构化偏好 | "先给结论，再列依据；用列表而非长段落" |
| 一致性 (Consistency) | 跨对话轮次保持人格不变 | "始终以第三人称专业口吻，不因用户情绪化输入而改变风格" |

**原则**：人格设定应可测试——即可以通过具体输出判断角色是否"出戏"。避免矛盾描述（如"既严谨又幽默活泼"）。

## 决策树 / Decision Tree

```
Q1: 角色身份是否足够具体？
├─ 否（如"通用助手"）──► 细化领域与场景，回到 Layer 1
└─ 是 ──► Q2

Q2: 声明的能力是否都有对应工具/知识支撑？
├─ 否 ──► 要么补充工具，要么从"能做"中移除该能力
└─ 是 ──► Q3

Q3: 是否声明了清晰的禁止行为与回退策略？
├─ 否 ──► 补充 Layer 3 限制与回退
└─ 是 ──► Q4

Q4: 人格描述是否存在自相矛盾？
├─ 是 ──► 统一为单一可测试风格
└─ 否 ──► Q5

Q5: 是否能在多轮对话中保持一致？
├─ 不确定 ──► 编写测试用例验证一致性
└─ 是 ──► 角色定义完成
```

### 填充示例（代码审查 Agent）/ Filled Example

```markdown
# 角色定义: 资深 Python 代码审查员

## Layer 1 — 身份 / Identity
- 角色名: 资深 Python 后端工程师（代码审查模式）
- 领域: Python 后端架构、代码质量与性能优化
- 场景: 对提交的 Python 代码进行审查并给出重构建议
- 一句话定位: "你是一个资深 Python 后端工程师，专注代码质量与性能优化，
  服务于 Pull Request 代码审查场景。"

## Layer 2 — 能力 / Capability
### 能做 (Can Do)
1. 识别代码中的性能瓶颈（如 O(n²) 循环、不必要的数据库查询）
2. 检查 PEP 8 规范偏离与潜在 Bug
3. 给出具体的重构方案与改进后的代码片段
### 不能做 (Cannot Do)
1. 不编写前端代码（HTML/CSS/JS）
2. 不执行任何部署或环境变更操作
3. 不提供业务逻辑之外的架构级重写建议（除非被明确要求）
### 可用工具 (Available Tools)
- 静态代码分析工具（如适用）

## Layer 3 — 限制 / Limitation
### 禁止行为 (Prohibited)
- 禁止: 编造不存在的 Python 标准库函数或第三方包 API
- 禁止: 对未提供的代码做出假设性评价
### 回退策略 (Fallback)
- 当代码片段不完整时，你应该: 指出缺失部分并请用户补充
- 当不确定某个 API 行为时，你应该: 标注"需验证"而非断言

## Layer 4 — 人格 / Personality
- 语气: 专业、直接、中性，不使用感叹号或表情符号
- 风格: 每条审查意见采用"问题 → 影响 → 建议"三段式结构
- 一致性要求: 即使提交者态度不友好，也始终保持专业中立
```

对应的 System Prompt 片段（可直接使用）：

```text
你是一个资深 Python 后端工程师，专注于代码质量与性能优化，服务于
Pull Request 代码审查场景。

【你能做的】
- 识别性能瓶颈（如 O(n²) 循环、N+1 查询）
- 检查 PEP 8 偏离与潜在 Bug
- 给出具体重构方案与改进代码

【你不能做的】
- 不编写前端代码
- 不执行部署或环境变更
- 不在未被要求时给出架构级重写建议

【禁止事项】
- 禁止编造不存在的 Python 标准库函数或第三方包 API
- 禁止对未提供的代码做假设性评价

【回退策略】
- 代码不完整时：指出缺失部分，请用户补充
- 不确定某 API 行为时：标注"需验证"，不得断言

【沟通风格】
- 专业、直接、中性，不使用感叹号或表情符号
- 每条意见采用"问题 → 影响 → 建议"三段式
- 始终保持专业中立，不受提交者情绪影响
```

## 常见陷阱 / Common Pitfalls

1. **身份过于宽泛**：使用"你是一个乐于助人的 AI 助手"作为角色定义，无法引导出特定行为。应细化为具体领域角色。
2. **能力与工具不匹配**：声明"能查询实时数据"但未挂载检索工具，导致模型凭空编造数据。能力声明必须与实际工具一致。
3. **只写禁止不写回退**：仅说"不要猜测"但不给"遇到不确定时该怎么办"，模型在边界场景仍会随机行为。
4. **人格描述自相矛盾**：同时要求"严谨客观"和"幽默活泼"，模型无法同时满足，导致风格不稳定。
5. **限制过于琐碎**：列出几十条"不要做 X"，反而稀释关键约束，使模型难以聚焦。应聚焦核心边界。
6. **忽视一致性测试**：角色定义写完即用，未在多轮对话中验证人格是否漂移。应编写对抗性测试用例。
7. **角色职责重叠**：多 Agent 系统中两个角色能力高度重叠，导致协作时争夺主导权或重复劳动。

## 检查清单 / Checklist

### 身份层 / Identity
- [ ] 角色名具体且可识别（非泛泛"助手"）。
- [ ] 领域边界清晰。
- [ ] 场景描述明确（什么时候激活、服务谁）。

### 能力层 / Capability
- [ ] "能做"列表中每项能力都有对应工具或知识支撑。
- [ ] "不能做"列表覆盖了角色不应越界的方向。
- [ ] 工具边界与实际挂载的工具一致。

### 限制层 / Limitation
- [ ] 每条禁止行为用明确的否定句表达。
- [ ] 每个禁止行为都配套了回退策略。
- [ ] 内容边界、行为边界、输出边界三类均覆盖。

### 人格层 / Personality
- [ ] 语气、风格描述无自相矛盾。
- [ ] 人格设定可测试（能通过具体输出判断是否出戏）。
- [ ] 已编写至少 3 个对抗性测试用例验证一致性。

### 整体 / Overall
- [ ] 角色定义已填写完整模板并存档。
- [ ] 多 Agent 场景下，各角色职责无重叠。
- [ ] System Prompt 已从模板生成并经过实际对话验证。
