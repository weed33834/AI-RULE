# 用户测试指南 (User Testing Guide)

> 本文档提供智能体用户测试的系统化方法：3 阶段测试流程、测试用例设计、反馈收集与分析。
> 自动化测试验证功能正确性，用户测试验证体验可用性。
> 与 `evaluation-framework.md` 互补——评估框架用指标度量，用户测试用真人验证。
> 与 `agent-testing-automation.md` 互补——自动化测试覆盖回归，用户测试发现盲区。

## §1 核心原则

- **真人真场景**：用户测试的核心价值在于发现自动化测试覆盖不到的盲区。
- **任务导向**：不问"你觉得怎么样"，问"请完成这个任务"。
- **观察优于询问**：用户做的时候遇到的问题 > 用户事后说的问题。
- **小批量多次**：5 个用户测 3 轮 > 15 个用户测 1 轮。每轮改进后再测。

## §2 三阶段测试流程

```
┌──────────────┐
│ 阶段1: 内测   │  开发者自己 + 团队成员测试，发现明显问题
└──────┬───────┘
       ▼
┌──────────────┐
│ 阶段2: 友好测 │  5-8 位目标用户，发现体验问题和边缘场景
└──────┬───────┘
       ▼
┌──────────────┐
│ 阶段3: 公测   │  更大范围用户，验证稳定性和可扩展性
└──────────────┘
```

### 2.1 阶段一：内测（Alpha）

| 维度 | 内容 |
|------|------|
| 参与者 | 开发者 + 团队成员（1-3 人） |
| 目标 | 发现功能性 bug、明显的设计缺陷 |
| 任务 | 覆盖所有核心功能路径 |
| 方法 | 自由探索 + 指定任务 |
| 产出 | Bug 列表 + 快速修复 |
| 通过标准 | 无 P0 bug，核心路径全部可完成 |

### 2.2 阶段二：友好测试（Beta）

| 维度 | 内容 |
|------|------|
| 参与者 | 目标用户画像匹配的真人用户（5-8 人） |
| 目标 | 发现体验问题、理解偏差、边缘场景 |
| 任务 | 3-5 个真实场景任务（非功能测试） |
| 方法 | 任务测试 + 放声思考 + 事后访谈 |
| 产出 | 体验问题列表 + 改进建议 + 满意度评分 |
| 通过标准 | 80% 用户能独立完成任务，满意度 ≥ 4/5 |

### 2.3 阶段三：公测（Gamma）

| 维度 | 内容 |
|------|------|
| 参与者 | 更大范围用户（20+ 人） |
| 目标 | 验证稳定性、性能、可扩展性 |
| 任务 | 自由使用 + 数据收集 |
| 方法 | 后台监控 + 问卷 + 留存率分析 |
| 产出 | 性能报告 + 留存数据 + 功能请求列表 |
| 通过标准 | 错误率 < 5%，日留存 > 40% |

## §3 测试用例设计

### 3.1 任务设计原则

| 原则 | 说明 | 示例 |
|------|------|------|
| 真实性 | 任务来自真实使用场景 | "帮我查一下上周的销售数据" 而非 "请调用 query_sales API" |
| 完整性 | 任务有明确的完成标准 | "找到最便宜的商品并告诉我价格" |
| 渐进性 | 从简单到复杂排列 | 先单轮对话，再多轮对话 |
| 开放性 | 不暗示具体操作路径 | "我想了解今天的天气" 而非 "用天气工具查天气" |

### 3.2 标准任务模板

```
任务卡片模板：
─────────────────────
任务描述: [用户视角的自然语言描述]
背景设定: [用户的身份和情境]
成功标准: [什么算完成]
预计轮次: [预期需要几轮对话]
关注点:   [测试者重点观察的维度]
─────────────────────
```

示例：
```
任务描述: "我想给团队订下周三的会议室，大概 10 个人，需要投影仪"
背景设定: 你是一个不太懂技术的部门经理，第一次使用这个助手
成功标准: 成功预订到满足条件的会议室，或告知无可用会议室并推荐替代方案
预计轮次: 3-5 轮
关注点:   意图理解准确性、多轮对话能力、信息补全方式
```

### 3.3 边缘场景设计

| 场景类型 | 示例 | 测试目的 |
|----------|------|----------|
| 模糊输入 | "帮我弄一下那个东西" | 意图澄清能力 |
| 矛盾输入 | "订一个 5 人的 3 人会议室" | 矛盾检测能力 |
| 超范围请求 | "帮我删除数据库" | 安全边界 |
| 中途改主意 | "不对，我改主意了，换成周四" | 上下文切换 |
| 极端输入 | 粘贴 5000 字文本 | 长输入处理 |
| 静默/无响应 | 用户不回复 | 超时处理 |

## §4 放声思考法

让用户在完成任务时说出自己的想法：

```
测试者指南（给用户）：
"请在做任务的时候，把你脑子里的想法说出来。
包括：你在期待什么、你看到了什么、你困惑什么、你想做什么。
不用解释为什么，直接说出来就好。"

测试者观察记录：
- 用户在哪里停顿了
- 用户说了什么困惑的话
- 用户尝试了什么没成功的操作
- 用户是否表达了不满或惊讶
```

### 4.1 观察记录模板

```
观察记录表：
时间 | 用户行为 | 用户言语 | 测试者备注
-----|---------|---------|----------
0:15 | 输入第一句话 | "我先试试问它..." | 输入自然
0:45 | 等待回复 | "怎么这么慢..." | 响应延迟 3s
1:02 | 看到回复 | "这个格式看不太懂" | 输出格式问题
1:30 | 追问 | "我换个说法试试" | 第一轮理解失败
```

## §5 事后访谈

任务完成后 5-10 分钟的半结构化访谈：

### 5.1 标准问题清单

| # | 问题 | 目的 |
|---|------|------|
| 1 | 刚才的任务你觉得完成了吗？ | 验证任务完成度感知 |
| 2 | 哪个环节让你觉得最顺畅？ | 识别优势 |
| 3 | 哪个环节让你最困惑或沮丧？ | 识别痛点 |
| 4 | 如果可以改一个地方，你改什么？ | 获取改进优先级 |
| 5 | 你平时会用类似的工具吗？ | 了解替代方案 |
| 6 | 1-5 分打几分？为什么不是满分/更低分？ | 满意度 + 改进线索 |

### 5.2 访谈原则

- 不引导：问"你觉得怎么样"而非"你觉得是不是太慢了"。
- 不辩护：用户说问题时，记录而非解释"这是因为..."。
- 追问细节：用户说"不太好"时，问"具体是哪里不好？"
- 记录原话：用户的话比测试者的总结更有价值。

## §6 反馈分析

### 6.1 反馈分类

| 类别 | 定义 | 处理优先级 |
|------|------|-----------|
| 阻塞问题 | 用户无法完成任务 | P0 — 立即修复 |
| 严重问题 | 用户能完成但非常困难 | P1 — 优先修复 |
| 一般问题 | 用户能完成但有困扰 | P2 — 排入迭代 |
| 建议改进 | 用户觉得可以更好 | P3 — 收集评估 |
| 正面反馈 | 用户觉得好的地方 | 记录为设计原则 |

### 6.2 问题归因分析

```
用户问题: "我不知道它为什么突然变了话题"

归因分析:
├── 是提示词问题？ → 检查提示词是否有规则冲突
├── 是上下文问题？ → 检查多轮对话是否丢失了关键上下文
├── 是工具问题？   → 检查工具返回是否异常
├── 是模型问题？   → 检查模型输出是否偏离指令
└── 是交互问题？   → 检查是否有明确的对话转折标记
```

## §7 测试报告模板

```
# 用户测试报告 v1.0

## 测试概况
- 日期: 2025-01-15
- 参与者: 6 人（符合目标用户画像）
- 任务数: 4 个
- 总用时: 3 小时

## 关键指标
| 指标 | 结果 | 目标 | 状态 |
|------|------|------|------|
| 任务完成率 | 75% (18/24) | 80% | ⚠️ |
| 平均完成时间 | 4分30秒 | 3分钟 | ⚠️ |
| 平均满意度 | 3.8/5 | 4.0/5 | ⚠️ |
| 需帮助次数 | 6次 | <5次 | ⚠️ |

## 主要发现
1. [P0] 3/6 用户在任务2中无法完成预订（意图理解失败）
2. [P1] 4/6 用户表示输出格式难以阅读
3. [P1] 2/6 用户在多轮对话中遇到上下文丢失

## 改进建议
1. 优化意图理解：增加澄清问题机制
2. 优化输出格式：采用结构化卡片格式
3. 优化上下文管理：增加每轮状态摘要
```

## §8 与其他文档的关系

- **`evaluation-framework.md`**: 评估框架提供量化指标，用户测试提供质性洞察。
- **`agent-testing-automation.md`**: 自动化测试是用户测试的前置——功能 bug 应在自动化阶段解决。
- **`prompt-iteration-guide.md`**: 用户测试发现的问题是提示词迭代的重要输入。
- **`conversation-design.md`**: 多轮对话体验问题参考该文档优化。
- **`evolution-policy.md`**: 用户测试反馈驱动智能体持续演进。
