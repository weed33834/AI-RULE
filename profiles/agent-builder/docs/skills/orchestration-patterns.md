# 编排模式目录 (Orchestration Patterns)

> 本文档定义多智能体系统的 5 种标准编排模式，源自 Azure 和 OpenAI 的共识框架。
> 与 `multi-agent.md` 互补——后者定义"如何设计角色"，本文档定义"如何编排协作"。

## §1 核心原则：最低可靠复杂度

**单次模型调用能解决就不要用 Agent；单 Agent 能解决就不要上多 Agent。**

```
问题复杂度评估
├─ 简单 → 单次模型调用（无 Agent）
├─ 需要工具 → 单 Agent + 工具循环
├─ 需要多角色 → 多 Agent（选本节模式）
└─ 需要确定性路由 → 优先确定性编排而非涌现式协商
```

**关键洞察**：研究显示，确定性编排（路由决策遵循稳定可复现规则）比纯涌现式 LLM 间协商在生产中准确率更高、推理成本更低。

## §2 五种编排模式

### 2.1 顺序编排 (Sequential / Pipeline)

```
Agent A → Agent B → Agent C → 输出
```

| 维度 | 说明 |
|------|------|
| 机制 | 预定义线性流水线，前一个 Agent 的输出是后一个的输入 |
| 适用场景 | 渐进式精炼：草稿→审查→润色、翻译→校对→排版 |
| 优势 | 简单、可预测、易调试 |
| 劣势 | 无并行、延迟累加、一个节点失败全链路中断 |
| 容错 | 每个节点设超时和降级输出 |

### 2.2 并发编排 (Concurrent / Fan-out Fan-in)

```
           ┌→ Agent A ─┐
输入 ──────┼→ Agent B ─┼─→ 聚合 → 输出
           └→ Agent C ─┘
```

| 维度 | 说明 |
|------|------|
| 机制 | 多个 Agent 并行处理同一任务，结果聚合 |
| 适用场景 | 多视角分析、投票决策、多方案对比 |
| 优势 | 并行加速、多视角交叉验证 |
| 劣势 | 资源消耗大、聚合逻辑复杂 |
| 容错 | 部分失败时用可用结果聚合，标注缺失 |

### 2.3 群聊编排 (Group Chat)

```
Agent A ←→ Agent B
   ↕           ↕
Agent C ←→ Agent D
```

| 维度 | 说明 |
|------|------|
| 机制 | 多个 Agent 以对等方式参与，自由交互 |
| 适用场景 | 头脑风暴、协作设计、开放讨论 |
| 优势 | 涌现式创意、灵活交互 |
| 劣势 | 难以控制方向、token 消耗高、易跑题 |
| 容错 | 设最大轮次限制 + 主持人 Agent 介入 |

### 2.4 交接编排 (Handoff / Decentralized)

```
Agent A → (交接) → Agent B → (交接) → Agent C
         传递上下文      传递上下文
```

| 维度 | 说明 |
|------|------|
| 机制 | Agent 之间动态交接控制权，系统记录最后活跃的 Agent |
| 适用场景 | 客服路由（技术→账单→物流）、多阶段工作流 |
| 优势 | 动态路由、上下文传递、用户面对单一入口 |
| 劣势 | 交接逻辑复杂、上下文可能丢失 |
| 容错 | 交接失败时回退到上一个 Agent |

### 2.5 管理器模式 (Manager / Agents-as-Tools)

```
            ┌→ 调用专家 Agent A
管理器 Agent ─┼→ 调用专家 Agent B
            └→ 调用专家 Agent C
     ↑
  用户交互
```

| 维度 | 说明 |
|------|------|
| 机制 | 中央管理 Agent 通过工具调用协调专家 Agent，保留上下文与控制权 |
| 适用场景 | 只需一个 Agent 面对用户的场景、复杂任务分解 |
| 优势 | 统一入口、上下文不丢失、专家可独立优化 |
| 劣势 | 管理器成为瓶颈、专家间无法直接通信 |
| 容错 | 专家失败时管理器降级处理或切换备选专家 |

## §3 选型决策树

```
用户需要面对几个 Agent？
├─ 1 个 → 管理器模式（Manager）
│   └─ 任务需要分解给专家 → Manager + 专家 Agent 作为工具
│   └─ 任务可单 Agent 完成 → 单 Agent（不需要多 Agent）
└─ 多个 → 
    任务有固定顺序吗？
    ├─ 是 → 顺序编排（Sequential）
    └─ 否 → 
        需要多视角/投票吗？
        ├─ 是 → 并发编排（Concurrent）
        └─ 否 → 
            需要动态路由吗？
            ├─ 是 → 交接编排（Handoff）
            └─ 否 → 群聊编排（Group Chat）

框架选型参考：
├─ 需要精细循环/分支/HITL/容错 → LangGraph
├─ 需要快速角色化多 Agent 原型 → CrewAI
├─ Azure 企业生态 → Microsoft Agent Framework (MAF)
└─ 简单 handoff + 内置 tracing + guardrail → OpenAI Agents SDK
```

## §4 编排层架构

生产级多 Agent 系统应分层关注：

```
┌─────────────────────────────────┐
│ 编排层 (Orchestrator)            │  ← 神经系统
│  ├─ 接收任务、分解子任务           │
│  ├─ 选择 Worker、监控完成          │
│  ├─ 持有终止条件和递归限制          │
│  └─ 错了则下游全部继承失败          │
├─────────────────────────────────┤
│ Worker 层                        │  ← 执行单元
│  ├─ 窄领域专家（research/code/data）│
│  ├─ 各自有 system prompt 和工具集  │
│  └─ 各自有失败模式                 │
├─────────────────────────────────┤
│ 工具与记忆层                      │  ← 最容易被低估
│  ├─ 工具：schema 校验 + 幂等执行   │
│  ├─ 记忆：短期 checkpointer + 长期 │
│  └─ 大多数生产故障的源头           │
└─────────────────────────────────┘
```

## §5 上下文膨胀控制

每个 Agent 加入自己的推理/工具结果/中间输出，上下文快速增长。

| 控制策略 | 说明 |
|----------|------|
| Agent 间压缩 | 交接时只传摘要，不传完整推理过程 |
| 共享状态外部化 | 共享状态持久化到外部存储，不依赖内存上下文 |
| 上下文修剪 | 定期选择性修剪非关键上下文 |
| 递归限制 | 编译图设 `recursion_limit`，防止无限循环 |
