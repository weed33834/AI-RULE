# 提示词迭代指南 (Prompt Iteration Guide)

> 本文档提供系统化的提示词迭代方法论：5 步迭代循环、A/B 测试模式、回归测试策略。
> 提示词不是一次性写完的，而是通过持续迭代逼近最优。
> 与 `prompt-patterns.md` 互补——模式提供正向参考，本文档提供改进流程。
> 与 `evaluation-framework.md` 互补——评估框架度量效果，本文档驱动改进。

## §1 核心理念

- **提示词是代码**：需要版本控制、测试、code review、持续迭代。
- **改一处可能崩三处**：提示词修改有蝴蝶效应，必须回归测试。
- **数据驱动迭代**：不靠"感觉更好了"，靠评估指标说话。
- **小步迭代**：一次只改一个变量，测量效果，再决定下一步。

## §2 五步迭代循环

```
┌─────────────┐
│ 1. 基线测量  │  用当前提示词跑评估集，记录基线指标
└──────┬──────┘
       ▼
┌─────────────┐
│ 2. 假设设计  │  基于失败案例分析，提出改进假设
└──────┬──────┘
       ▼
┌─────────────┐
│ 3. 变更实施  │  只改与假设相关的部分，保持其他不变
└──────┬──────┘
       ▼
┌─────────────┐
│ 4. 效果测量  │  用相同评估集跑新提示词，对比指标
└──────┬──────┘
       ▼
┌─────────────┐
│ 5. 决策保留  │  指标提升 → 保留；下降 → 回退；持平 → 换方向
└─────────────┘
```

### 2.1 第一步：基线测量

在修改提示词前，必须先建立基线：

```
基线记录模板：
- 提示词版本: v1.3.0
- 评估集: eval-set-v2 (50 条)
- 指标:
  - 准确率: 72% (36/50)
  - 格式合规率: 88% (44/50)
  - 平均响应长度: 342 tokens
  - 拒答率: 4% (2/50)
- 失败案例: #07, #13, #22, #29, #31, #38, #41, #44, #47, #49
- 高频失败模式: 格式错误(4次)、信息遗漏(3次)、幻觉(2次)、过度回答(1次)
```

### 2.2 第二步：假设设计

基于失败案例，提出具体、可验证的改进假设：

| 假设格式 | 示例 |
|----------|------|
| 现象 → 原因 → 改法 → 预期效果 | 格式错误(4次) → 输出格式说明不够醒目 → 用 XML 标签包裹格式要求 → 格式合规率提升到 95%+ |

**好的假设**：具体、可验证、有预期指标。
**坏的假设**："让提示词更好一点"、"优化一下措辞"。

### 2.3 第三步：变更实施

- 一次只改一个变量。如果同时改 3 处，无法知道哪处起了作用。
- 保留旧版本（git 版本控制），便于回退对比。
- 在提示词头部标注变更记录。

### 2.4 第四步：效果测量

用与基线完全相同的评估集和指标测量：

```
对比报告模板：
| 指标 | 基线(v1.3.0) | 新版(v1.3.1) | 变化 |
|------|-------------|-------------|------|
| 准确率 | 72% | 80% | +8% ↑ |
| 格式合规率 | 88% | 96% | +8% ↑ |
| 平均响应长度 | 342 | 358 | +16 |
| 拒答率 | 4% | 2% | -2% ↓ |

新通过案例: #13, #22, #29, #38
新失败案例: #15 (之前通过，现在失败)
净改善: +3 案例
```

### 2.5 第五步：决策保留

| 结果 | 决策 | 后续行动 |
|------|------|----------|
| 全面提升 | 保留新版本 | 记录成功经验，寻找下一个改进点 |
| 部分提升部分下降 | 评估权衡 | 如果净收益为正可保留，否则尝试混合方案 |
| 全面下降 | 回退到基线 | 分析失败原因，换方向重新假设 |
| 持平 | 换方向 | 当前假设无效，尝试其他改进方向 |

## §3 A/B 测试模式

当不确定哪个改法更好时，同时测试多个版本：

```
评估集 (50条)
├── 版本 A (当前) → 准确率 72%
├── 版本 B (改法1: 加 few-shot) → 准确率 78%
└── 版本 C (改法2: 加 CoT) → 准确率 81%

→ 选择版本 C，但分析 B 在哪些案例上更好，考虑融合
```

### 3.1 A/B 测试注意事项

- 评估集必须有足够的样本量（建议 ≥ 30 条）。
- 评估集应覆盖典型场景和边缘场景。
- 评估标准必须客观（不靠人感判断）。
- 多版本测试时注意 token 成本。

## §4 回归测试策略

提示词修改的蝴蝶效应：改了输出格式，可能影响推理质量。

### 4.1 必须回归测试的场景

| 变更类型 | 回归测试范围 |
|----------|------------|
| 修改输出格式 | 全量评估（格式变化可能影响下游解析） |
| 修改角色定义 | 全量评估（角色变化影响所有行为） |
| 修改单条规则 | 聚焦测试该规则相关案例 + 抽样 20% 其他案例 |
| 添加新规则 | 测试新规则相关案例 + 确认不影响现有案例 |
| 修改示例(few-shot) | 全量评估（示例影响模型行为模式） |

### 4.2 回归测试集构建

```
评估集结构建议：
├── 核心场景 (40%) — 最常见的用户请求
├── 边缘场景 (30%) — 不常见但重要的场景
├── 对抗场景 (15%) — 试图绕过规则的输入
├── 格式测试 (10%) — 验证输出格式合规性
└── 回归保护 (5%) — 之前修复过的失败案例
```

## §5 常见迭代模式

### 5.1 格式问题迭代

| 问题 | 迭代策略 |
|------|----------|
| 输出格式不稳定 | 用 XML 标签包裹格式要求，加 "必须" 强调 |
| JSON 解析失败 | 加示例 JSON，明确禁止 markdown 代码块包裹 |
| 输出过长 | 加字数限制 + "简洁" 强调 + few-shot 短示例 |

### 5.2 准确性问题迭代

| 问题 | 迭代策略 |
|------|----------|
| 信息遗漏 | 加检查清单"回答前确认以下信息已包含" |
| 幻觉 | 加"不确定时回答不知道" + 来源标注要求 |
| 过度回答 | 加"只回答被问到的" + 反例展示 |

### 5.3 行为一致性问题迭代

| 问题 | 迭代策略 |
|------|----------|
| 角色漂移 | 在多处重复角色定义（头部 + 工作流中 + 自检中） |
| 规则遗忘 | 加每轮自检清单"检查是否违反了以下规则" |
| 风格不一致 | 加 few-shot 示例展示期望风格 |

## §6 版本管理规范

```
提示词版本号: MAJOR.MINOR.PATCH

- MAJOR: 角色定义或核心工作流变更（不兼容变更）
- MINOR: 新增规则或修改现有规则（兼容变更）
- PATCH: 修改措辞、示例、格式微调（行为不变）

变更记录格式：
## v1.4.0 (2025-01-15)
### Added
- 新增 <error_recovery> 规则：异常时输出结构化错误报告
### Changed
- <output_format> 增加 markdown 表格支持
### Fixed
- 修复 few-shot 示例中参数顺序错误（#22, #29 通过）
```
